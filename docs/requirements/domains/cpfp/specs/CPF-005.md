# CPF-005: Package Fee Rate Computation

> **Authoritative requirement:** [CPF-005](../NORMATIVE.md#CPF-005)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 5.8

## Summary

Package fee rates aggregate the economic value of an entire CPFP dependency chain into a single metric. A child transaction that pays a high fee effectively "pays for" its low-fee parent, because including the child in a block requires including the parent first. The package fee rate captures this by summing fees and costs across the entire ancestor chain.

This enables the mempool to make correct economic decisions: a low-fee parent with a high-fee child should not be evicted, and the child should be prioritized based on the aggregate chain's economics, not just its own fee.

Package fee rates are computed during Phase 2 after dependency resolution (CPF-002). They are stored in the `MempoolItem` and used by:
- Block selection (Section 6): items sorted by `package_fee_per_virtual_cost_scaled`.
- Descendant score (CPF-006): ancestors' protection scores incorporate descendants' package FPC.
- Eviction ordering: `descendant_score` (derived from package FPC) determines eviction priority.

This is a dig-mempool extension. Chia L1 does not compute package fee rates because it does not support CPFP.

## Specification

### Computation

For an item with `depends_on` (set of parent bundle IDs):

```rust
let ancestor_fees: u64 = depends_on.iter()
    .map(|parent_id| items[parent_id].package_fee)
    .sum();
let ancestor_costs: u64 = depends_on.iter()
    .map(|parent_id| items[parent_id].package_virtual_cost)
    .sum();

item.package_fee = item.fee + ancestor_fees;
item.package_virtual_cost = item.virtual_cost + ancestor_costs;
item.package_fee_per_virtual_cost_scaled =
    (item.package_fee as u128 * FPC_SCALE) / item.package_virtual_cost as u128;
```

**Important:** Ancestor aggregation uses each parent's `package_fee` and `package_virtual_cost` (not just `fee` and `virtual_cost`). This ensures transitive ancestors are included: if C depends on B depends on A, then C's package includes A's fee even though A is not a direct parent.

### No-Dependency Case

For items with empty `depends_on`:
```rust
item.package_fee = item.fee;
item.package_virtual_cost = item.virtual_cost;
item.package_fee_per_virtual_cost_scaled = item.fee_per_virtual_cost_scaled;
```

### MempoolItem Fields

```rust
pub package_fee: u64,
pub package_virtual_cost: u64,
pub package_fee_per_virtual_cost_scaled: u128,
```

## Acceptance Criteria

- [ ] `package_fee` equals `fee + sum(ancestor.package_fee)` for items with dependencies.
- [ ] `package_virtual_cost` equals `virtual_cost + sum(ancestor.package_virtual_cost)` for items with dependencies.
- [ ] `package_fee_per_virtual_cost_scaled` equals `(package_fee * FPC_SCALE) / package_virtual_cost`.
- [ ] Items with no dependencies have `package_fee == fee`, `package_virtual_cost == virtual_cost`.
- [ ] Transitive ancestors are included via parents' `package_*` values.
- [ ] All arithmetic uses `u128` for FPC computation (no floating-point).
- [ ] Division by zero is handled (package_virtual_cost should never be 0 but must not panic).

## Implementation Notes

- **Primary file:** `src/mempool/cpfp.rs`
- **Ordering:** Computed after CPF-002 (dependency resolution) and CPF-003/CPF-004 (depth/cycle checks).
- **Multiple parents:** When an item has multiple direct parents, their `package_fee` and `package_virtual_cost` values are summed. This correctly accounts for all ancestors even when the dependency graph is a DAG (not just a chain).
- **Overlap handling:** If two parents share a common ancestor, that ancestor's fee is counted twice. This is intentional and conservative — the package FPC is an upper bound on the chain's effective FPC. Deduplicating shared ancestors would require expensive graph traversal and provides marginal benefit.
- **FPC_SCALE:** `1_000_000_000_000` (u128) from SPEC Section 2.5.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| No dependencies | Unit test | Item with no parents | package_fee == fee; package_virtual_cost == virtual_cost |
| Single parent chain | Unit test | P (fee=100, vc=1000) -> C (fee=200, vc=500) | C: package_fee=300, package_virtual_cost=1500 |
| Multi-level chain | Unit test | A (fee=10) -> B (fee=20) -> C (fee=30) | C: package_fee=60 (A+B+C); similar for costs |
| Multiple parents | Unit test | P1 (fee=50, vc=100) and P2 (fee=100, vc=200); C depends on both | C: package_fee = C.fee + 50 + 100; package_vc = C.vc + 100 + 200 |
| FPC scaled correctly | Unit test | fee=1000, virtual_cost=500 | package_fpc_scaled = 1000 * FPC_SCALE / 500 = 2 * FPC_SCALE |
| Transitive ancestors included | Unit test | A -> B -> C (3-level); verify C includes A's fee | C.package_fee includes A.fee via B.package_fee |
| Zero fee item in chain | Unit test | P (fee=0) -> C (fee=500) | C.package_fee=500; C.package_fpc reflects total |
| Division safety | Unit test | Edge case with very small virtual_cost values | No panic; correct scaled result |
