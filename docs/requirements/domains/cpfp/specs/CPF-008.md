# CPF-008: Cross-Bundle Announcement Validation

> **Authoritative requirement:** [CPF-008](../NORMATIVE.md#CPF-008)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 5.9

---

## Summary

When a CPFP child bundle asserts announcements or messages that are created by its ancestor bundles, those assertions cannot be validated during intra-bundle CLVM execution (handled by `dig-clvm`), because the creating and asserting conditions live in different `SpendBundle`s. The mempool must perform cross-bundle announcement validation during Phase 2 to verify that these inter-bundle assertions are satisfiable.

Three types of cross-bundle assertions are validated:
1. **ASSERT_COIN_ANNOUNCEMENT**: The child asserts an announcement hash that must match a `CREATE_COIN_ANNOUNCEMENT` from an ancestor bundle.
2. **ASSERT_PUZZLE_ANNOUNCEMENT**: The child asserts a puzzle announcement hash that must match a `CREATE_PUZZLE_ANNOUNCEMENT` from an ancestor bundle.
3. **RECEIVE_MESSAGE**: The child receives a message that must match a `SEND_MESSAGE` from an ancestor bundle.

Announcement IDs are computed using the `announcement_id()` function from `chia-sdk-types`, which computes `sha256(coin_info || message)` for coin announcements and `sha256(puzzle_hash || message)` for puzzle announcements.

Only assertions that reference ancestor bundles (bundles in the `depends_on` chain) are validated. Assertions referencing non-ancestor bundles are NOT rejected -- they may be satisfied by other bundles in the same block during block validation, which is outside the mempool's scope.

This is a dig-mempool extension (Improvement #13 in SPEC Section 1.4). Chia L1 does not validate cross-bundle announcements because it does not support CPFP.

## Specification

### Scope

This validation applies ONLY to items with non-empty `depends_on` (CPFP items). Items with no dependencies skip this step entirely.

### Algorithm

1. **Collect ancestor announcements:** For each ancestor bundle in the dependency chain (transitive), collect:
   - `created_coin_announcements`: Set of `announcement_id(coin_id, message)` from `CREATE_COIN_ANNOUNCEMENT` conditions.
   - `created_puzzle_announcements`: Set of `announcement_id(puzzle_hash, message)` from `CREATE_PUZZLE_ANNOUNCEMENT` conditions.
   - `sent_messages`: Set of message identifiers from `SEND_MESSAGE` conditions.

2. **Check child assertions:** For each assertion condition in the new bundle:
   - `ASSERT_COIN_ANNOUNCEMENT(hash)`: Check if `hash` is in `created_coin_announcements` from ancestors OR if it was already satisfied intra-bundle (by dig-clvm).
   - `ASSERT_PUZZLE_ANNOUNCEMENT(hash)`: Check if `hash` is in `created_puzzle_announcements` from ancestors OR if it was already satisfied intra-bundle.
   - `RECEIVE_MESSAGE(msg)`: Check if `msg` matches a `sent_messages` from ancestors OR was satisfied intra-bundle.

3. **Non-ancestor assertions:** Assertions that reference neither intra-bundle nor ancestor conditions are NOT rejected. They are left for block-level validation.

### announcement_id Computation

From `chia-sdk-types`:
```rust
// Coin announcement: sha256(coin_id || message)
let coin_ann_id = announcement_id(coin_id, message);

// Puzzle announcement: sha256(puzzle_hash || message)
let puzzle_ann_id = announcement_id(puzzle_hash, message);
```

### Condition Sources

Assertions and creations are read from `OwnedSpendBundleConditions.spends[*]`:
- `CREATE_COIN_ANNOUNCEMENT`: from per-spend conditions.
- `ASSERT_COIN_ANNOUNCEMENT`: from per-spend conditions.
- `CREATE_PUZZLE_ANNOUNCEMENT`: from per-spend conditions.
- `ASSERT_PUZZLE_ANNOUNCEMENT`: from per-spend conditions.
- `SEND_MESSAGE`: from per-spend conditions.
- `RECEIVE_MESSAGE`: from per-spend conditions.

## Acceptance Criteria

- [ ] Cross-bundle announcement validation runs only for CPFP items (non-empty `depends_on`).
- [ ] `ASSERT_COIN_ANNOUNCEMENT` is checked against ancestors' `CREATE_COIN_ANNOUNCEMENT`.
- [ ] `ASSERT_PUZZLE_ANNOUNCEMENT` is checked against ancestors' `CREATE_PUZZLE_ANNOUNCEMENT`.
- [ ] `RECEIVE_MESSAGE` is checked against ancestors' `SEND_MESSAGE`.
- [ ] `announcement_id()` from `chia-sdk-types` is used for hash computation.
- [ ] Assertions satisfied intra-bundle (by dig-clvm) are not re-checked or flagged.
- [ ] Assertions referencing non-ancestor bundles are NOT rejected.
- [ ] Ancestor conditions include transitive ancestors (not just direct parents).

## Implementation Notes

- **Primary file:** `src/mempool/cpfp.rs` or `src/mempool/announcements.rs`
- **Ordering:** Runs in Phase 2 after dependency resolution (CPF-002) and before conflict detection (CFR-001). SPEC pipeline step 9.
- **Intra-bundle vs cross-bundle:** dig-clvm validates all intra-bundle announcements during Phase 1. This step only needs to validate assertions that could not be satisfied within the child bundle itself. In practice, the simplest implementation collects all ancestor announcements and checks all unsatisfied assertions from the child.
- **Performance:** Collecting ancestor announcements requires iterating all ancestors' conditions. With `max_dependency_depth = 25` and typical condition counts, this is fast.
- **SPEC note:** "Assertions referencing non-ancestor bundles are not rejected -- they may be satisfied by other bundles in the same block during block validation (outside mempool scope)." This means the validation is best-effort: it confirms ancestor announcements are present but does not reject bundles with unsatisfied non-ancestor assertions.
- **Condition access:** Conditions are available in `MempoolItem.conditions` (`OwnedSpendBundleConditions`), which is stored for every active item.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| Coin announcement satisfied by parent | Unit test | Parent creates coin announcement A; child asserts A | Child admitted |
| Puzzle announcement satisfied by parent | Unit test | Parent creates puzzle announcement B; child asserts B | Child admitted |
| RECEIVE_MESSAGE satisfied by parent | Unit test | Parent sends message M; child receives M | Child admitted |
| Coin announcement from grandparent | Unit test | Grandparent creates announcement; grandchild asserts it | Grandchild admitted (transitive ancestor) |
| Unsatisfied ancestor assertion not rejected | Unit test | Child asserts announcement not from any ancestor (may be from sibling) | Child admitted (non-ancestor assertions not rejected) |
| No validation for non-CPFP items | Unit test | Item with no dependencies that asserts announcements | No cross-bundle validation performed |
| announcement_id correctly computed | Unit test | Verify announcement_id matches chia-sdk-types output | Hashes match |
| Multiple announcements from multiple ancestors | Unit test | Parent1 creates ann A; Parent2 creates ann B; child asserts both A and B | Child admitted |
