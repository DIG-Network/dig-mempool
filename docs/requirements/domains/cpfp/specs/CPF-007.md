## Summary

Cascade eviction is the mechanism that maintains CPFP consistency when a parent item is removed from the mempool. When a parent is removed (for any reason: block confirmation, RBF replacement, capacity eviction, explicit removal, or expiry), all of its dependents become invalid because the coins they spend (created by the parent) no longer exist in the mempool. These dependents must be recursively removed.

The cascade is depth-first and recursive: if removing parent A causes child B to be evicted, and B has its own child C, then C is also evicted. All indexes must be cleaned for every evicted item: `items`, `coin_index`, `mempool_coins`, `dependencies`, `dependents`, `dedup_index`, and `singleton_spends`.

After cascade eviction completes, ancestor `descendant_score` values must be recomputed for any ancestors of the evicted items that remain in the mempool (CPF-006).

This is a dig-mempool extension. Chia L1 does not support CPFP and therefore has no cascade eviction logic.

## Specification

### Algorithm

```
cascade_evict(bundle_id):
    item = items[bundle_id]

    // 1. Recursively evict all dependents first (depth-first)
    if dependents[bundle_id] exists:
        for child_id in dependents[bundle_id].clone():
            cascade_evict(child_id)

    // 2. Clean all indexes for this item
    for removal in item.removals:
        coin_index.remove(&removal)
    for addition in item.additions:
        mempool_coins.remove(&addition.coin_id())
    for parent_id in &item.depends_on:
        dependents[parent_id].remove(&bundle_id)
    dependencies.remove(&bundle_id)
    dependents.remove(&bundle_id)
    // Also clean dedup_index, singleton_spends if applicable

    // 3. Update accumulators
    total_cost -= item.cost
    total_fees -= item.fee

    // 4. Remove from items
    items.remove(&bundle_id)

    // 5. Fire event hook
    fire on_item_removed(bundle_id, RemovalReason::CascadeEvicted { parent_id })
```

### Post-Cascade Cleanup

After all cascade evictions complete:
1. Walk up ancestor chains of any remaining items that were siblings of evicted items.
2. Recompute `descendant_score` for affected ancestors (CPF-006).

### RemovalReason

```rust
RemovalReason::CascadeEvicted { parent_id: Bytes32 }
```

The `parent_id` is the direct parent whose removal triggered this item's cascade eviction, not the root cause.

## Acceptance Criteria

- [ ] Removing a parent recursively removes all direct and transitive dependents.
- [ ] All indexes are cleaned for every evicted item: `items`, `coin_index`, `mempool_coins`, `dependencies`, `dependents`.
- [ ] `dedup_index` and `singleton_spends` are cleaned if applicable.
- [ ] Cost and fee accumulators are decremented for each evicted item.
- [ ] `on_item_removed` fires with `RemovalReason::CascadeEvicted` for each evicted dependent.
- [ ] Ancestor `descendant_score` values are recomputed after cascade eviction.
- [ ] The eviction does not panic on already-removed items (idempotent removal).
- [ ] All evicted bundle IDs are collected and returned to the caller.

## Implementation Notes

- **Primary file:** `src/mempool/eviction.rs`
- **Recursion safety:** Maximum recursion depth is bounded by `max_dependency_depth` (default 25), so stack overflow is not a concern. Alternatively, an iterative approach using a stack or queue is acceptable.
- **Clone before iterate:** When iterating `dependents[bundle_id]` and recursively evicting, the set must be cloned first (or iterated via a collected Vec) to avoid borrow conflicts during removal.
- **Shared with CFR-006:** The cascade eviction algorithm is shared between CPFP general removal (this requirement) and RBF replacement (CFR-006). The only difference is the initial `RemovalReason` for the root item.
- **Trigger contexts:** Cascade eviction is triggered by:
  - Block confirmation (`on_new_block` with `RemovalReason::Confirmed` for root)
  - RBF replacement (CFR-006 with `RemovalReason::ReplacedByFee` for root)
  - Capacity eviction (with `RemovalReason::CapacityEviction` for root)
  - Expiry (with `RemovalReason::Expired` for root)
  - Explicit removal (with `RemovalReason::ExplicitRemoval` for root)
  - Clear (with `RemovalReason::Cleared` for all)

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| Single child eviction | Unit test | P creates coin X; C spends X. Remove P. | Both P and C removed |
| Multi-level cascade | Unit test | P -> C1 -> C2 -> C3. Remove P. | All 4 items removed |
| Sibling preservation | Unit test | P creates coins X and Y; C1 spends X; C2 spends Y. Remove C1. | Only C1 removed; P and C2 remain |
| Index cleanup complete | Unit test | After cascade, verify coin_index, mempool_coins, dependencies, dependents | No stale entries for evicted items |
| Accumulator correctness | Unit test | Track total_cost before/after cascade of 3 items | Reduced by sum of 3 items' costs |
| Descendant score recomputed | Unit test | P has children C1 (high FPC) and C2 (low FPC). Remove C1. | P.descendant_score recomputed from C2 only |
| Event hooks fire for each | Unit test | Register hook; cascade-evict 3 items | on_item_removed called 3 times with CascadeEvicted |
| Diamond dependency cascade | Unit test | P1 creates X; P2 creates Y; C spends both X and Y. Remove P1. | C cascade-evicted; P2 remains but loses C as dependent |
