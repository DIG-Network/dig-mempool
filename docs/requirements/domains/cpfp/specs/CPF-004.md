# CPF-004: Defensive Cycle Detection

> **Authoritative requirement:** [CPF-004](../NORMATIVE.md#CPF-004)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

In the UTXO (coinset) model, dependency cycles should be structurally impossible: a coin can only be created once and spent once, so a circular dependency (A depends on B depends on A) would require A to spend a coin created by B while B spends a coin created by A. Since coins are identified by `sha256(parent_coin_id || puzzle_hash || amount)`, creating such a cycle would require finding a hash collision, which is computationally infeasible.

Nevertheless, the mempool performs a defensive cycle check after inserting dependency edges. This guards against implementation bugs in the dependency resolution logic, corrupted state from snapshot restore, or unforeseen edge cases. If a cycle is detected, the bundle is rejected with `MempoolError::DependencyCycle`.

The cycle check is lightweight because the dependency graph is bounded by `max_dependency_depth` (CPF-003), and the check only needs to traverse the ancestor chain of the newly inserted item to see if it encounters itself.

---

## Specification

### Algorithm

After recording dependency edges (CPF-002) for the new bundle:

1. Starting from the new bundle, follow `dependencies` edges (parent direction).
2. At each node, check if the current node is the new bundle itself.
3. If the new bundle is encountered during traversal -> `MempoolError::DependencyCycle`.
4. If traversal reaches a node with no dependencies (depth 0) without encountering the new bundle -> no cycle.

The traversal is bounded by `max_dependency_depth` iterations at most.

### Error Type

```rust
MempoolError::DependencyCycle
```

### Alternative: DFS Cycle Detection

A full DFS from the new bundle through both `dependencies` and `dependents` edges is also acceptable, though more expensive. The simpler ancestor-only traversal is sufficient because:
- Cycles require the new bundle to be reachable from itself via parent edges.
- Existing items were already cycle-checked when they were inserted.
- The only new edges are those involving the new bundle.

---

## Acceptance Criteria

- [ ] After inserting dependency edges, a cycle check is performed.
- [ ] If a cycle is detected, `MempoolError::DependencyCycle` is returned.
- [ ] The dependency edges inserted for the rejected bundle are rolled back (no partial graph state).
- [ ] In normal UTXO operation, the cycle check never triggers (vacuously passes).
- [ ] The cycle check completes in O(max_dependency_depth) time.
- [ ] The cycle check does not panic or loop infinitely even on corrupted graph state.

---

## Implementation Notes

- **Primary file:** `src/mempool/cpfp.rs`
- **Ordering:** Cycle detection runs immediately after CPF-002 records dependency edges and CPF-003 checks depth.
- **Rollback:** If a cycle is detected, the edges just inserted into `dependencies` and `dependents` must be removed before returning the error. This ensures the graph remains in a valid state.
- **Why defensive:** In thousands of hours of Chia L1 operation, no UTXO cycle has ever been observed. This check exists purely as a safety net for implementation correctness.
- **Visited set:** Use a `HashSet<Bytes32>` to track visited nodes during traversal, preventing infinite loops on corrupted graph state where a non-self cycle exists among ancestors.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| No cycle in linear chain | Unit test | P -> C1 -> C2 (linear chain) | No cycle detected; all items admitted |
| No cycle with multiple parents | Unit test | P1 and P2 both create coins; C spends both | No cycle detected |
| Synthetic cycle detected | Unit test | Manually corrupt dependency graph to create A -> B -> A cycle; submit C depending on A | DependencyCycle error |
| Rollback on cycle detection | Unit test | Trigger cycle detection; verify dependency graph is clean | No stale edges from rejected bundle |
| Bounded traversal | Unit test | Chain at max depth; verify cycle check completes without timeout | Completes in O(max_dependency_depth) |
| Empty dependencies no cycle | Unit test | Item with no dependencies | Cycle check passes trivially |
| Diamond dependency no cycle | Unit test | P creates coins X and Y; C1 spends X; C2 spends Y; G spends coins from both C1 and C2 | No cycle; G depends on both C1 and C2 |
