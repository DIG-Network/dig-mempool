# CPF-006: Descendant Score Tracking

> **Authoritative requirement:** [CPF-006](../NORMATIVE.md#CPF-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 5.17

## Summary

The descendant score protects low-fee parent transactions from eviction when they have valuable children. Without this mechanism, a low-fee parent could be evicted during capacity pressure, cascade-evicting its high-fee children and losing significant aggregate fee revenue from the mempool.

Each `MempoolItem` maintains a `descendant_score` that equals the maximum of:
1. Its own `fee_per_virtual_cost_scaled` (individual FPC).
2. The `package_fee_per_virtual_cost_scaled` of any descendant in its dependency chain.

The score is updated dynamically:
- **On child addition:** Walk up the ancestor chain from the new child, updating each ancestor's `descendant_score` to `max(current_score, child.package_fpc_scaled)`.
- **On child removal:** Walk up the ancestor chain and recompute `descendant_score` from remaining descendants.

The `descendant_score` is used as the primary sort key for eviction ordering: items with lower descendant scores are evicted first. This is an improvement over Chia L1, which sorts by raw `fee_per_cost` for eviction.

## Specification

### MempoolItem Field

```rust
/// Maximum of (own FPC, any descendant chain's package FPC).
pub descendant_score: u128,
```

### Initial Value

On insertion, before any descendants exist:
```rust
item.descendant_score = item.fee_per_virtual_cost_scaled;
```

### Update on Child Addition

When a new item is inserted with dependencies:
```rust
for parent_id in &new_item.depends_on {
    let mut current_id = *parent_id;
    loop {
        let current_item = items.get_mut(&current_id).unwrap();
        let new_score = max(
            current_item.descendant_score,
            new_item.package_fee_per_virtual_cost_scaled,
        );
        if new_score == current_item.descendant_score {
            break;  // No change; no need to propagate further
        }
        current_item.descendant_score = new_score;
        // Continue to current_item's parents
        if current_item.depends_on.is_empty() {
            break;
        }
        // Walk to next ancestor (for multi-parent, walk all)
        for &grandparent_id in &current_item.depends_on {
            // Propagate to grandparent...
        }
    }
}
```

### Recomputation on Child Removal

When a child is removed (eviction, confirmation, cascade):
```rust
fn recompute_descendant_score(item_id: Bytes32) {
    let item = items.get(&item_id).unwrap();
    let mut score = item.fee_per_virtual_cost_scaled;
    if let Some(children) = dependents.get(&item_id) {
        for child_id in children {
            let child = items.get(child_id).unwrap();
            score = max(score, child.package_fee_per_virtual_cost_scaled);
        }
    }
    items.get_mut(&item_id).unwrap().descendant_score = score;
    // Propagate up to ancestors
}
```

## Acceptance Criteria

- [ ] `descendant_score` is initialized to `fee_per_virtual_cost_scaled` on insertion.
- [ ] When a child is added, ancestors' `descendant_score` is updated to `max(current, child.package_fpc)`.
- [ ] Score propagation stops when no change occurs (optimization).
- [ ] When a child is removed, affected ancestors' `descendant_score` is recomputed from remaining descendants.
- [ ] `descendant_score` is used as the primary eviction ordering key (lower score evicted first).
- [ ] Items with no descendants have `descendant_score == fee_per_virtual_cost_scaled`.

## Implementation Notes

- **Primary file:** `src/mempool/cpfp.rs` and `src/mempool/eviction.rs`
- **Performance:** Score propagation on addition is O(depth) per parent path and terminates early when no change occurs. Recomputation on removal examines all remaining direct children, which is typically a small number.
- **Mutability:** `descendant_score` is the only mutable field on an otherwise-immutable `MempoolItem`. Consider using `Cell<u128>` or maintaining it as a separate map (`HashMap<Bytes32, u128>`) to avoid requiring `&mut` access to the item.
- **Multi-parent propagation:** When an item has multiple parents, the score update propagates along each ancestor path independently.
- **Chia comparison:** Chia L1 sorts eviction candidates by `fee_per_cost` directly (no descendant score). This means Chia can evict a low-fee parent even if it has valuable children. dig-mempool's descendant score prevents this.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| Initial score no children | Unit test | Insert item with no dependencies | descendant_score == fee_per_virtual_cost_scaled |
| Score updated on child add | Unit test | Parent (FPC=10); add child (package FPC=50) | Parent descendant_score becomes 50 |
| Score not downgraded | Unit test | Parent (FPC=10, descendant_score=50); add child with package FPC=30 | Parent descendant_score remains 50 |
| Score recomputed on child remove | Unit test | Parent with child (package FPC=50); remove child | Parent descendant_score drops to own FPC |
| Multi-level propagation | Unit test | A -> B -> C; C has high package FPC | A.descendant_score reflects C's package FPC |
| Early termination | Unit test | Add child that does not increase any ancestor's score | Propagation stops at first unchanged ancestor |
| Eviction respects descendant score | Integration test | Two items: low own FPC with high-FPC child vs medium own FPC alone; trigger eviction | Medium-FPC item evicted first (lower descendant_score) |
| Recomputation with multiple children | Unit test | Parent with children C1 (FPC=30) and C2 (FPC=50); remove C2 | Parent descendant_score becomes max(own FPC, 30) |
