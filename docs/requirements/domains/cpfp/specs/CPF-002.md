## Summary

Dependency resolution is the process by which the mempool determines whether a newly submitted bundle depends on other active mempool items. During Phase 2 (under write lock), for each coin the new bundle spends (each removal), the mempool first checks if the coin is in the caller-provided `coin_records`. If not, it falls back to the `mempool_coins` index (CPF-001) to see if the coin was created by an existing active mempool item.

If the coin is found in `mempool_coins`, a dependency edge is recorded: the new bundle depends on the creating bundle. If the coin is not found in either `coin_records` or `mempool_coins`, the bundle is rejected with `MempoolError::CoinNotFound`.

The dependency graph is maintained as two complementary maps: `dependencies` (bundle -> set of parent bundles) and `dependents` (bundle -> set of child bundles). These enable efficient traversal for cascade eviction, package fee computation, and block selection.

This is a dig-mempool extension not present in Chia L1.

## Specification

### Data Structures

```rust
// bundle_id -> set of parent bundle_ids
dependencies: HashMap<Bytes32, HashSet<Bytes32>>
// bundle_id -> set of child bundle_ids
dependents: HashMap<Bytes32, HashSet<Bytes32>>
```

### Algorithm

For each `removal` in the new bundle's removals:
1. Check `coin_records.contains_key(&removal)`:
   - If yes, this is an on-chain coin. No CPFP dependency.
2. If not in `coin_records`, check `mempool_coins.get(&removal)`:
   - If `Some(parent_bundle_id)`: record dependency edge.
     - `dependencies[new_bundle_id].insert(parent_bundle_id)`
     - `dependents[parent_bundle_id].insert(new_bundle_id)`
   - If `None`: return `Err(MempoolError::CoinNotFound(removal))`.

After processing all removals:
- `item.depends_on = dependencies[new_bundle_id].clone()` (or equivalent).
- Compute `item.depth` = 1 + max depth of any parent.

### MempoolItem Fields

```rust
/// Bundle IDs this item directly depends on.
pub depends_on: HashSet<Bytes32>,
/// Depth in the dependency chain. 0 = no dependencies.
pub depth: u32,
```

## Acceptance Criteria

- [ ] Coins present in `coin_records` do not create dependency edges.
- [ ] Coins not in `coin_records` but in `mempool_coins` create a dependency edge.
- [ ] Coins not in `coin_records` and not in `mempool_coins` cause `MempoolError::CoinNotFound`.
- [ ] Both `dependencies` and `dependents` maps are updated atomically.
- [ ] `item.depends_on` reflects all direct parent bundle IDs.
- [ ] `item.depth` is computed as 1 + max(parent depths), or 0 if no dependencies.
- [ ] A bundle can depend on multiple parent bundles (spending coins from different parents).

---

## Implementation Notes

- **Primary file:** `src/mempool/cpfp.rs` or within Phase 2 admission logic.
- **Ordering:** Dependency resolution runs after Phase 1 validation and before conflict detection (CFR-001). The SPEC pipeline (Section 5.1) places CPFP resolution at step 8, before conflict detection at step 10.
- **Multiple parents:** A bundle may depend on multiple parents if it spends coins created by different active items. Each parent is recorded as a separate dependency edge.
- **Depth computation:** For multiple parents, depth = 1 + max(parent.depth for parent in depends_on). This is computed by looking up each parent's `MempoolItem.depth` value.
- **No on-chain coins in mempool_coins:** `mempool_coins` only contains coins created by active mempool items, not on-chain coins. This ensures no confusion between on-chain and unconfirmed coins.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| On-chain coin no dependency | Unit test | Bundle spends coin in coin_records | depends_on is empty; depth = 0 |
| Mempool coin creates dependency | Unit test | Parent creates coin X; child spends X (not in coin_records) | Child depends_on contains parent ID; depth = 1 |
| CoinNotFound for missing coin | Unit test | Bundle spends coin not in coin_records or mempool_coins | MempoolError::CoinNotFound |
| Multiple parents | Unit test | Bundle spends coins from two different active parents | depends_on contains both parent IDs |
| Depth computation single chain | Unit test | P (depth=0) -> C1 (depth=1) -> C2 | C2.depth == 2 |
| Depth computation multiple parents | Unit test | P1 (depth=0) and P2 (depth=2); child spends coins from both | child.depth == 3 (1 + max(0, 2)) |
| Bidirectional graph consistency | Unit test | After inserting CPFP chain, verify dependencies and dependents match | For each (child, parent) in dependencies, parent appears in dependents |
| Mixed on-chain and mempool coins | Unit test | Bundle spends 2 on-chain coins and 1 mempool coin | depends_on contains only the mempool coin's creator |
