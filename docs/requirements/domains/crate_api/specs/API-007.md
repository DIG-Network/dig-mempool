## Summary

`AdmissionPolicy` and `BlockSelectionStrategy` are extension traits that allow callers to customize mempool behavior without modifying the core crate. They enable domain-specific logic for admission decisions and block construction priorities.

Both traits are dig-mempool extensions not present in Chia L1 (SPEC Section 14.2). Chia's mempool has hardcoded admission rules and a single selection algorithm. dig-mempool provides these hooks for DIG L2-specific requirements (e.g., rejecting certain puzzle types, prioritizing governance transactions).

## Specification

### AdmissionPolicy Trait

```rust
/// Admission policy: domain-specific validation after standard checks pass.
pub trait AdmissionPolicy {
    fn check(
        &self,
        item: &MempoolItem,
        existing_items: &[Arc<MempoolItem>],
    ) -> Result<(), String>;
}
```

**Semantics:**
- Called during Phase 2 of the admission pipeline (SPEC Section 5.16), after all standard checks (dedup, validation, conflict, RBF, capacity) have passed.
- `item` is the candidate `MempoolItem` about to be admitted.
- `existing_items` is a snapshot of all current active items (for cross-item policy decisions).
- Returns `Ok(())` to allow admission, or `Err(String)` to reject with `MempoolError::PolicyRejected`.
- The policy does NOT mutate the mempool. It is a pure decision function.

**Use Cases:**
- Reject bundles that create too many coins.
- Reject bundles involving specific puzzle hashes (blocklist).
- Rate-limit submissions from specific puzzle types.
- Enforce domain-specific business rules.

### BlockSelectionStrategy Trait

```rust
/// Custom block selection strategy.
pub trait BlockSelectionStrategy {
    fn select(
        &self,
        eligible_items: &[Arc<MempoolItem>],
        max_block_cost: u64,
        max_spends: usize,
    ) -> Vec<Arc<MempoolItem>>;
}
```

**Semantics:**
- Called by `select_for_block_with_strategy()` instead of the built-in 4-way greedy algorithm.
- `eligible_items` contains all active items that pass pre-selection filtering (no expired, no future timelocked).
- The implementation MUST:
  - Not exceed `max_block_cost` total cost.
  - Not exceed `max_spends` total spend count.
  - Not include conflicting items (shared spent coins).
  - Include all ancestors of any selected CPFP item.
  - Return items in topological order (parents before children).
- The implementation SHOULD optimize for total fees.

**Use Cases:**
- Prioritize governance transactions regardless of fee rate.
- Implement MEV-resistant ordering.
- Custom packing algorithms (e.g., knapsack).

- [ ] `AdmissionPolicy` trait is publicly exported.

- [ ] `BlockSelectionStrategy::select()` has the specified signature.
- [ ] `submit_with_policy()` invokes `AdmissionPolicy::check()`.
- [ ] `select_for_block_with_strategy()` invokes `BlockSelectionStrategy::select()`.
- [ ] `PolicyRejected` error is returned when policy rejects.
- [ ] Custom strategy output is returned directly by `select_for_block_with_strategy()`.

## Implementation Notes

- **Primary file:** `src/traits.rs`
- **Object safety:** Both traits must be object-safe (`dyn AdmissionPolicy`, `dyn BlockSelectionStrategy`). The signatures are already object-safe (no `Self` in return position, no generics).
- **Thread safety:** Traits are called under the write lock (policy) or read lock (strategy). Implementors should not perform blocking operations.
- **Strategy validation:** The mempool does NOT validate strategy output (e.g., check for conflicts or topological order). The strategy is trusted. Invalid output may cause block validation failures downstream.
