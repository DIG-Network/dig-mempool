## Summary

`FeeTracker` is the internal data structure that accumulates historical fee data from confirmed blocks and provides the statistical basis for fee rate estimation. It uses a bucket-based approach inspired by Bitcoin Core's `CBlockPolicyEstimator`, following Chia's implementation in `BitcoinFeeEstimator` ([`bitcoin_fee_estimator.py:14-78`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/bitcoin_fee_estimator.py#L14)) which wraps `FeeTracker` and `SmartFeeEstimator`.

Chia's `FeeEstimatorInterface` protocol ([`fee_estimator_interface.py:11-33`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/fee_estimator_interface.py#L11)) defines the methods: `new_block`, `add_mempool_item`, `remove_mempool_item`, `estimate_fee_rate`. dig-mempool internalizes these as methods on the `FeeTracker` struct rather than using a trait, since the implementation is fixed.

The tracker maintains logarithmically spaced fee-rate buckets and a circular buffer of per-block confirmed transaction data, enabling both instantaneous and historical fee analysis.

## Specification

### FeeTracker Struct

```rust
struct FeeTracker {
    /// Number of recent blocks to track. Default: 100.
    window: usize,
    /// Fee-rate buckets (logarithmically spaced from min to max FPC).
    buckets: Vec<FeeBucket>,
    /// Circular buffer of per-block confirmed transaction data.
    block_history: VecDeque<BlockFeeData>,
    /// Current height.
    current_height: u64,
}
```

### FeeBucket Struct

```rust
struct FeeBucket {
    /// Lower bound of this bucket's fee rate (scaled FPC).
    fee_rate_lower: u128,
    /// Upper bound.
    fee_rate_upper: u128,
    /// Number of transactions confirmed within 1 block at this rate.
    confirmed_in_1: f64,
    /// Within 2 blocks.
    confirmed_in_2: f64,
    /// Within 5 blocks.
    confirmed_in_5: f64,
    /// Within 10 blocks.
    confirmed_in_10: f64,
    /// Total transactions observed at this rate.
    total_observed: f64,
}
```

### BlockFeeData Struct

```rust
pub struct BlockFeeData {
    pub height: u64,
    pub min_fpc_included: u128,
    pub max_fpc_included: u128,
    pub median_fpc: u128,
    pub num_transactions: usize,
}
```

### Bucket Spacing

Buckets MUST be logarithmically spaced from a minimum fee rate (0 or 1 scaled FPC) to a maximum fee rate. The default number of buckets is 50 (configurable via `MempoolConfig::fee_estimator_buckets`). Logarithmic spacing ensures fine granularity at low fee rates (where most transactions cluster) and coarser granularity at high fee rates.

### Confirmation Tracking

Each bucket tracks how many transactions at that fee rate were confirmed within N blocks of entering the mempool, for N in {1, 2, 5, 10}. This data drives the confidence-based estimation in [FEE-003](FEE-003.md).

### Rolling Window

The `block_history` circular buffer retains data for the most recent `window` blocks (default: 100). When a new block is recorded and the buffer is full, the oldest entry is evicted.

## Acceptance Criteria

- [ ] `FeeTracker` struct exists with `window`, `buckets`, `block_history`, and `current_height` fields.
- [ ] Buckets are logarithmically spaced across the fee-rate range.
- [ ] Default bucket count is 50 (configurable).
- [ ] Default window size is 100 blocks (configurable).
- [ ] Each `FeeBucket` tracks `confirmed_in_1`, `confirmed_in_2`, `confirmed_in_5`, `confirmed_in_10`, and `total_observed`.
- [ ] `block_history` is a bounded circular buffer that evicts oldest entries.
- [ ] `BlockFeeData` captures per-block fee statistics.

## Implementation Notes

- **Primary file:** `src/fee/tracker.rs`
- **Internal only:** `FeeTracker` is not part of the public API. It is used internally by `Mempool` and exposed only through `estimate_fee_rate()` and `record_confirmed_block()`.
- **Thread safety:** Protected by `fee_lock: RwLock` (SPEC Section 8.9).
- **Bucket boundaries:** Use `10^(i * range / num_buckets)` for logarithmic spacing, where `range` spans from `log10(min_fpc)` to `log10(max_fpc)`.
- **f64 usage:** Confirmation counts use `f64` because exponential decay (FEE-004) produces fractional values. This is internal-only and does not affect deterministic mempool ordering.

| Custom bucket count | Unit test | Create with fee_estimator_buckets=20 | 20 logarithmically spaced buckets |
