## Summary

`record_confirmed_block(height, bundles)` feeds confirmed transaction fee/cost data from a newly confirmed block into the `FeeTracker`. This is the mechanism by which the fee estimator learns from actual on-chain confirmation behavior.

The function is called internally by `on_new_block()` as step 5 of the block confirmation lifecycle (SPEC Section 9.1). It is also exposed publicly for manual seeding (e.g., loading historical data on startup before the mempool has observed any blocks).

To prevent stale data from dominating the fee estimate, older observations are decayed using an exponential decay factor of 0.998 per block. This means data from 100 blocks ago retains approximately 82% weight (`0.998^100 = ~0.818`), and data from 500 blocks ago retains approximately 37% weight (`0.998^500 = ~0.368`).

This mirrors Chia's `BitcoinFeeEstimator.new_block()` ([`bitcoin_fee_estimator.py:34-36`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/bitcoin_fee_estimator.py#L34)) which calls `FeeTracker.process_block()` to update bucket statistics.

## Specification

### Function Signature

```rust
impl Mempool {
    pub fn record_confirmed_block(&self, height: u64, bundles: &[ConfirmedBundleInfo]);
}
```

### ConfirmedBundleInfo

```rust
pub struct ConfirmedBundleInfo {
    pub cost: u64,
    pub fee: u64,
    pub num_spends: usize,
}
```

### Recording Algorithm

1. Apply exponential decay factor (0.998) to all existing bucket counters:
   - For each bucket: `confirmed_in_N *= 0.998`, `total_observed *= 0.998`.
2. For each `ConfirmedBundleInfo`:
   - Compute `virtual_cost = cost + (num_spends * spend_penalty_cost)`.
   - Compute `fpc_scaled = (fee * FPC_SCALE) / virtual_cost`.
   - Find the bucket containing `fpc_scaled`.
   - Increment `total_observed` for that bucket.
   - Determine how many blocks the transaction waited for confirmation (if trackable) and increment the appropriate `confirmed_in_N` counters.
3. Compute `BlockFeeData` for this block (min/max/median FPC, transaction count).
4. Append to `block_history`. If `block_history.len() > window`, evict oldest entry.
5. Update `current_height`.

### Decay Factor

The decay factor `0.998` is applied once per block (per call to `record_confirmed_block`). This is applied multiplicatively to all `f64` counters in all buckets before recording the new block's data.

### Integration with on_new_block

`on_new_block()` calls `record_confirmed_block()` with the `confirmed_bundles` parameter passed by the caller. The caller computes `ConfirmedBundleInfo` from the confirmed block's transactions.

- [ ] Accepts `height: u64` and `bundles: &[ConfirmedBundleInfo]`.
- [ ] Applies exponential decay factor 0.998 to all bucket counters before recording new data.

- [ ] Appends `BlockFeeData` to the rolling window.

- [ ] Called automatically by `on_new_block()`.
- [ ] Callable manually for historical data seeding.

## Implementation Notes

- **Primary file:** `src/fee/tracker.rs`
- **Thread safety:** Writes `fee_lock` (RwLock write). Called under `on_new_block()`'s write lock path.
- **Decay precision:** Using `f64` for decay arithmetic. After many blocks, very old data approaches zero naturally.
- **Confirmation wait time:** Ideally, the tracker records how long each transaction waited in the mempool before confirmation. Since the mempool only knows its own admission history, transactions that were never in this mempool (confirmed externally) use a default wait of 1 block.
- **Edge cases:**
  - Empty `bundles` slice: still apply decay and record an empty `BlockFeeData`.
  - `cost = 0`: skip (avoid division by zero in FPC computation).
  - Height regression: warn but process (caller may be feeding historical data out of order).
