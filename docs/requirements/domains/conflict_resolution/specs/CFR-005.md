# CFR-005: Conflict Cache on RBF Failure

> **Authoritative requirement:** [CFR-005](../NORMATIVE.md#CFR-005)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 5.11, 8.6

---

## Summary

When a submitted bundle conflicts with an existing active item but fails the RBF evaluation (any of CFR-002, CFR-003, or CFR-004), the bundle is not simply discarded. Instead, the mempool adds it to the **conflict cache**, a bounded secondary storage for bundles that are valid but currently blocked by a conflicting active item. If the conflicting item is later removed (e.g., confirmed in a block, expired, or itself replaced), the conflict cache item can be retried.

This matches Chia L1's behavior where bundles that fail RBF are stored in the `ConflictTxCache` for later retry. On Chia, `add_spend_bundle()` catches the conflict, attempts RBF, and on failure stores the bundle in `_conflict_cache` before returning the error to the caller.

The conflict cache has bounded capacity controlled by `MempoolConfig.max_conflict_count` (default 1,000) and `MempoolConfig.max_conflict_cost` (default 1x block cost). If the cache is full, the bundle is simply dropped with no further action.

---

## Specification

### Data Structure

```rust
// Conflict cache: bundle_id -> raw SpendBundle
conflict_cache: HashMap<Bytes32, SpendBundle>
```

### Algorithm

When RBF evaluation fails (any of CFR-002, CFR-003, CFR-004):
1. Check conflict cache capacity:
   - `conflict_cache.len() < config.max_conflict_count`
   - `conflict_cache_cost + bundle_cost <= config.max_conflict_cost`
2. If both conditions met:
   - Insert `(bundle_id, spend_bundle)` into `conflict_cache`.
   - Update `conflict_cache_cost`.
   - Fire `on_conflict_cached` event hook.
3. Return the appropriate `MempoolError` (regardless of cache insertion success).

### Capacity Defaults

```rust
pub const DEFAULT_MAX_CONFLICT_COUNT: usize = 1_000;
// max_conflict_cost defaults to 1x block cost
```

### Conflict Cache Lifecycle

- **Drain:** During `on_new_block()`, conflict cache items whose conflicting active item was removed are collected into `RetryBundles::conflict_retries` for caller resubmission.
- **Eviction:** No LRU or priority eviction — if the cache is full, new items are silently dropped.
- **Dedup:** Before adding to conflict cache, check `conflict_cache.contains_key(&bundle_id)` to avoid duplicates.

---

## Acceptance Criteria

- [ ] On RBF failure, the bundle is added to the conflict cache if space permits.
- [ ] The appropriate `MempoolError` variant (RbfNotSuperset, RbfFpcNotHigher, or RbfBumpTooLow) is still returned to the caller.
- [ ] Conflict cache respects `max_conflict_count` and `max_conflict_cost` limits.
- [ ] When the cache is full, the bundle is silently dropped (no error beyond the original RBF error).
- [ ] Duplicate bundle IDs are not inserted into the conflict cache.
- [ ] The `on_conflict_cached` event hook fires when a bundle is added to the cache.
- [ ] Conflict cache items are returned in `RetryBundles::conflict_retries` when their conflicting item is removed during `on_new_block()`.

## Implementation Notes

- **Primary file:** `src/mempool/conflict.rs` or `src/mempool/conflict_cache.rs`
- **Lock:** `conflict_lock` (RwLock) protects `conflict_cache` and `conflict_cache_cost`.
- **Raw SpendBundle storage:** The conflict cache stores raw `SpendBundle`s (not `MempoolItem`s) because items are re-validated on retry. This matches Chia's `ConflictTxCache` which stores `SpendBundle` objects.
- **Cost tracking:** The conflict cache tracks aggregate cost to prevent memory exhaustion from many large bundles.
- **Seen cache interaction:** Bundles in the conflict cache should NOT be rejected by the seen-cache on resubmission. Either the conflict cache bypasses the seen-cache, or the drain process removes the bundle ID from the seen-cache before returning it for retry.

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| RBF failure caches bundle | Unit test | Submit bundle that fails RBF superset check; verify it is in conflict cache | Bundle in conflict cache; RbfNotSuperset error returned |
| RBF failure returns error | Unit test | Submit conflicting bundle; verify error is returned even though cache insertion succeeds | Correct MempoolError variant returned |
| Cache respects count limit | Unit test | Fill conflict cache to max_conflict_count; submit another failing RBF | New bundle silently dropped; original error returned |
| Cache respects cost limit | Unit test | Fill conflict cache to max_conflict_cost; submit another failing RBF | New bundle silently dropped; original error returned |
| Duplicate not re-cached | Unit test | Submit same bundle twice (both fail RBF); verify only one entry in cache | conflict_cache.len() == 1 |
| Event hook fires on cache | Unit test | Register event hook; trigger RBF failure | `on_conflict_cached` called with bundle ID |
| Drain on new_block | Integration test | Cache a conflicting bundle; confirm the conflicting active item in a block; call on_new_block | Bundle appears in RetryBundles::conflict_retries |
| Cache cleared on clear() | Unit test | Add items to conflict cache; call clear() | Conflict cache is empty |
