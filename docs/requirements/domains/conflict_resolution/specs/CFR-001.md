# CFR-001: Conflict Detection via coin_index

> **Authoritative requirement:** [CFR-001](../NORMATIVE.md#CFR-001)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 5.10

---

## Summary

Conflict detection determines whether a newly submitted bundle attempts to spend coins that are already spent by an existing active mempool item. This is the entry point for all Replace-by-Fee (RBF) logic: if no conflicts are detected, the bundle is admitted directly; if conflicts exist, the RBF evaluation pipeline (CFR-002 through CFR-004) is invoked.

The mempool maintains a `coin_index` (`HashMap<Bytes32, Bytes32>`) mapping each spent coin ID to the bundle ID that spends it. For each removal (spent coin) in the new bundle, the mempool looks up this index. If a match is found, the existing bundle is recorded as a conflicting bundle. All unique conflicting bundle IDs are collected for subsequent RBF evaluation.

This mirrors Chia L1's `check_removals()` function, which queries the in-memory SQLite `spends` table to detect conflicts. On Chia, the `spends` table maps `coin_id` to `tx_id` (bundle ID), serving the same role as `coin_index` in dig-mempool.

Only the **active pool** participates in conflict detection. Pending (timelocked) items and conflict cache items are not checked, matching Chia's behavior where `check_removals()` only queries the active mempool's `spends` table.

---

## Specification

### Data Structure

```rust
// Active pool index: coin_id -> spending bundle_id
coin_index: HashMap<Bytes32, Bytes32>
```

### Algorithm

For each `removal` in the new bundle's removals:
1. Look up `coin_index.get(&removal)`.
2. If `Some(existing_bundle_id)` is returned, add `existing_bundle_id` to the conflict set.
3. If `None`, the coin is not spent by any active item (no conflict for this coin).

After iterating all removals:
- If the conflict set is empty, proceed to insertion (no RBF needed).
- If the conflict set is non-empty, proceed to RBF evaluation (CFR-002 through CFR-004).

### Scope

- Active pool items: checked via `coin_index`.
- Pending pool items: NOT checked (pending items do not participate in conflict detection).
- Conflict cache items: NOT checked (conflict cache stores raw `SpendBundle`s, not indexed coins).

---

## Acceptance Criteria

- [ ] For each removal in the submitted bundle, the mempool checks `coin_index` for an existing spending bundle.
- [ ] All unique conflicting bundle IDs are collected into a conflict set.
- [ ] Pending pool items do not appear in conflict detection results.
- [ ] Conflict cache items do not appear in conflict detection results.
- [ ] An empty conflict set results in direct admission (no RBF evaluation).
- [ ] A non-empty conflict set triggers RBF evaluation (CFR-002 through CFR-004).

---

## Implementation Notes

- **Primary file:** `src/mempool/conflict.rs` (or within the Phase 2 admission logic)
- **Data structure:** `coin_index` is populated at insertion time (one entry per removal in the admitted item) and cleaned up on removal/eviction.
- **Design Decision #2** from SPEC Section 1.3: "Pending items do NOT participate in active conflict detection — matches Chia L1."
- **Performance:** HashMap lookups are O(1) amortized. The number of removals per bundle is typically small (1-10 coins), making this step very fast under the write lock.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| No conflict on empty pool | Unit test | Submit a bundle to an empty mempool; check conflict detection returns empty set | Empty conflict set; bundle admitted |
| Single conflict detected | Unit test | Submit bundle A spending coin X, then bundle B also spending coin X | Conflict set contains bundle A's ID |
| Multiple conflicts detected | Unit test | Bundle A spends coin X, bundle B spends coin Y; bundle C spends both X and Y | Conflict set contains both A and B IDs |
| Pending items not detected | Unit test | Add a timelocked item spending coin X to pending pool; submit new bundle spending coin X | Empty conflict set; no conflict with pending item |
| Conflict cache not detected | Unit test | Add a bundle to conflict cache; submit new bundle spending same coin | Empty conflict set; conflict cache items are not indexed |
| Self-conflict impossible | Unit test | Submit a bundle; verify its own removals are indexed but not flagged during its own insertion | Bundle admitted without self-conflict |
| coin_index cleanup on removal | Unit test | Add bundle spending coin X, then remove it; submit new bundle spending coin X | Empty conflict set after removal cleanup |
| Multiple removals one conflict | Unit test | Bundle with 5 removals where only 1 overlaps with an existing bundle | Conflict set contains exactly 1 bundle ID |
