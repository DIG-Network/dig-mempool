# CFR-006: RBF + CPFP Cascade Eviction

> **Authoritative requirement:** [CFR-006](../NORMATIVE.md#CFR-006)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 5.11, 9.2

---

## Summary

When RBF successfully replaces a parent item that has CPFP dependents, the replacement fundamentally invalidates those dependents: their input coins (created by the replaced parent) no longer exist in the mempool. All dependents must be recursively removed (cascade eviction).

This is a dig-mempool extension not present in Chia L1 (Chia does not support CPFP dependency chains). The interaction between RBF and CPFP introduces a critical safety invariant: cascade-evicted children MUST NOT be added to the conflict cache, because their input coins are irrecoverably invalid (the creating bundle was replaced, and the replacement creates different output coins). Storing them in the conflict cache would waste space and cause guaranteed re-validation failures on retry.

The cascade eviction process is depth-first and recursive: if item A is replaced, its child B is evicted, and B's child C is also evicted, and so on. All evicted items must have their entries cleaned from `coin_index`, `mempool_coins`, the dependency graph, and all other indexes.

---

## Specification

### Algorithm

When RBF passes all three checks (CFR-002, CFR-003, CFR-004) and the conflicting items are to be replaced:

1. For each `conflicting_item` being replaced:
   a. Look up `dependents[conflicting_item.spend_bundle_id]`.
   b. For each dependent, recursively cascade-evict (depth-first):
      - Remove from `items`.
      - Remove all entries from `coin_index` (for its removals).
      - Remove all entries from `mempool_coins` (for its additions).
      - Remove from `dependencies` and `dependents` maps.
      - Remove from `dedup_index` if applicable.
      - Remove from `singleton_spends` if applicable.
      - Subtract cost and fee from accumulators.
      - Fire `on_item_removed` with `RemovalReason::CascadeEvicted { parent_id }`.
   c. Remove the conflicting item itself (same cleanup as above, but with `RemovalReason::ReplacedByFee { replacement_id }`).
2. After all evictions, walk the ancestor chains of any affected items and recompute `descendant_score`.
3. Insert the replacement item.
4. Record all cascade-evicted bundle IDs for reporting.

### Conflict Cache Exclusion

Cascade-evicted children MUST NOT be added to the conflict cache:
- Their input coins no longer exist (the parent was replaced).
- Adding them to the conflict cache would waste capacity.
- On retry, they would fail with `CoinNotFound`.

### Event Reporting

- Replaced items: `RemovalReason::ReplacedByFee { replacement_id }`
- Cascade-evicted dependents: `RemovalReason::CascadeEvicted { parent_id }`
- All cascade-evicted bundle IDs are available via event hooks and are included in `RetryBundles::cascade_evicted` during `on_new_block()` if the eviction happens during block processing.

## Acceptance Criteria

- [ ] When RBF replaces a parent, all direct and transitive dependents are recursively removed.
- [ ] Cascade-evicted items are NOT added to the conflict cache.
- [ ] All indexes are cleaned: `items`, `coin_index`, `mempool_coins`, `dependencies`, `dependents`, `dedup_index`, `singleton_spends`.
- [ ] Cost and fee accumulators are decremented for all evicted items.
- [ ] `on_item_removed` fires with `RemovalReason::CascadeEvicted` for each dependent.
- [ ] `on_item_removed` fires with `RemovalReason::ReplacedByFee` for the directly replaced item.
- [ ] Ancestor `descendant_score` values are recomputed after cascade eviction.
- [ ] The replacement item is inserted after all evictions are complete.

## Implementation Notes

- **Primary file:** `src/mempool/eviction.rs` or `src/mempool/conflict.rs`
- **Recursion depth:** The maximum dependency depth is bounded by `config.max_dependency_depth` (default 25), so recursion depth is bounded and stack overflow is not a concern. Iterative depth-first traversal is also acceptable.
- **Ordering:** Eviction must complete before insertion of the replacement. This ensures `coin_index` does not contain stale entries when the replacement's removals are indexed.
- **Descendant score recomputation:** After cascade eviction, ancestors of evicted items may have inflated `descendant_score` values. These must be recomputed by walking up ancestor chains and taking the max of remaining descendants' package FPC values.
- **Multi-level cascades:** If item A (replaced) has child B, and B has child C, the cascade order is: C evicted first (depth-first), then B, then A is replaced. This ensures no dangling references at any point.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| Single child cascade | Unit test | Parent P creates coin X; child C spends X (CPFP). RBF replaces P. | C is cascade-evicted; P is replaced |
| Multi-level cascade | Unit test | P -> C1 -> C2 (3-level chain). RBF replaces P. | C2 and C1 both cascade-evicted |
| Cascade not in conflict cache | Unit test | After cascade eviction, check conflict cache | Cascade-evicted items not in conflict cache |
| Replaced item not in conflict cache | Unit test | After successful RBF, check conflict cache for the replaced item | Replaced item not in conflict cache |
| Index cleanup verified | Unit test | After cascade eviction, verify coin_index, mempool_coins, dependencies, dependents are clean | All entries removed for evicted items |
| Descendant score recomputed | Unit test | Parent with descendant_score boosted by evicted child; verify recomputation | descendant_score drops to own FPC after child eviction |
| Event hooks fire correctly | Unit test | Register event hook; trigger RBF with cascade | CascadeEvicted reason for children; ReplacedByFee for parent |
| Accumulator decremented | Unit test | Track total_cost and total_fees before/after cascade | Accumulators reduced by evicted items' cost and fee |
