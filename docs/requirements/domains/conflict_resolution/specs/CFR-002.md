# CFR-002: RBF Superset Rule

> **Authoritative requirement:** [CFR-002](../NORMATIVE.md#CFR-002)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 5.11

---

## Summary

The superset rule is the first of three RBF conditions that must be satisfied for a new bundle to replace one or more existing conflicting bundles. It ensures that the replacement bundle spends at least all the same coins as every conflicting bundle it intends to replace. This prevents a scenario where a replacement bundle "frees" coins that the evicted bundles were spending, which could leave the mempool in an inconsistent state or enable certain DoS attacks.

On Chia L1, this is enforced in the `can_replace()` function: for every conflicting item, the function checks that all of the conflicting item's removals are present in the new item's removals. If any conflicting removal is missing from the new item, replacement is rejected.

When multiple bundles conflict with the new submission (e.g., the new bundle spends coins X and Y, where X is spent by bundle A and Y is spent by bundle B), the superset rule requires that the new bundle's removals include ALL removals from BOTH bundle A and bundle B.

---

## Specification

### Algorithm

Given:
- `new_removals`: `HashSet<Bytes32>` — the new bundle's removals.
- `conflicting_items`: `Vec<Arc<MempoolItem>>` — items identified by CFR-001.

For each `conflicting_item` in `conflicting_items`:
1. For each `removal` in `conflicting_item.removals`:
   - If `!new_removals.contains(&removal)` -> reject with `MempoolError::RbfNotSuperset`.

If all conflicting items' removals are contained in `new_removals`, proceed to CFR-003.

### Scope

The superset check applies only to the conflicting items' **own** removals. It does NOT include removals from CPFP dependents of the conflicting items. Dependents are handled separately by cascade eviction (CFR-006).

---

## Acceptance Criteria

- [ ] For each conflicting bundle, every removal must appear in the new bundle's removals.
- [ ] If any conflicting bundle has a removal not in the new bundle, `MempoolError::RbfNotSuperset` is returned.
- [ ] The check aggregates across all conflicting bundles (not just one).
- [ ] CPFP dependents' removals are NOT included in the superset check.
- [ ] On superset failure, no state mutation occurs (the failing bundle is not inserted).

---

## Implementation Notes

- **Primary file:** `src/mempool/conflict.rs` (RBF logic)
- **Data types:** `new_removals` should be pre-computed as a `HashSet<Bytes32>` for O(1) lookups. Conflicting items' removals are `Vec<Bytes32>` from `MempoolItem.removals`.
- **Edge case — single conflict:** When only one bundle conflicts, the superset check is straightforward: new bundle must include all removals of that one bundle.
- **Edge case — multiple conflicts:** When bundle A (spends X, Y) and bundle B (spends Y, Z) both conflict, new bundle must spend at least {X, Y, Z}. Note Y appears in both but is counted once.
- **Chia's implementation** iterates `conflicting_item.removals` and checks membership in the new bundle's removals set. dig-mempool follows the same approach.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| Exact match passes | Unit test | New bundle spends exactly the same coins as the conflicting bundle | Superset check passes |
| Strict superset passes | Unit test | New bundle spends the conflicting bundle's coins plus additional coins | Superset check passes |
| Missing removal fails | Unit test | Conflicting bundle spends coins {A, B}; new spends {A, C} (missing B) | `RbfNotSuperset` error |
| Multiple conflicts all satisfied | Unit test | Bundle A spends {X}, bundle B spends {Y}; new spends {X, Y, Z} | Superset check passes for both |
| Multiple conflicts one fails | Unit test | Bundle A spends {X}, bundle B spends {Y}; new spends {X, Z} (missing Y) | `RbfNotSuperset` error |
| CPFP dependent removals excluded | Unit test | Parent spends {A}, child spends {B} (CPFP); new bundle spends {A} only | Superset check passes (child's removals not checked) |
| Empty conflicting removals | Unit test | Conflicting bundle has zero removals (edge case) | Superset check passes trivially |
