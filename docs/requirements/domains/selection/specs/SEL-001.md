## Summary

`select_for_block()` is the primary entry point for block candidate selection. It accepts a cost budget, the current chain height, and the current timestamp, and returns an ordered set of non-conflicting mempool items suitable for block inclusion.

This function mirrors Chia's `create_bundle_from_mempool_items()` ([`mempool.py:583-615`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/mempool.py#L583)), which selects transactions for block building via a single `ORDER BY priority DESC, seq ASC` query ([`mempool.py:605`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/mempool.py#L605)). dig-mempool extends this with a multi-strategy greedy approach and CPFP awareness for better fee optimization.

The function reads from the active pool only (not pending or conflict caches). Items are returned as `Arc<MempoolItem>` for zero-copy sharing with the caller. The caller then passes the selected items to the block producer (e.g., `dig_clvm::build_block_generator()` for CLVM compression, `fast_forward_singleton()` for lineage rebasing, and signature aggregation). The mempool does not perform any block construction logic.

## Specification

### Function Signature

```rust
pub fn select_for_block(
    &self,
    max_block_cost: u64,
    height: u64,
    timestamp: u64,
) -> Vec<Arc<MempoolItem>>;
```

### Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `max_block_cost` | `u64` | Maximum total CLVM cost budget for the block. The selection will not exceed this. |
| `height` | `u64` | Current L2 chain height. Used to filter expired and future-timelocked items. |
| `timestamp` | `u64` | Current chain timestamp. Used to filter time-expired items. |

### Return Type

- `Vec<Arc<MempoolItem>>` -- Ordered list of selected items from the active pool. Never includes pending or conflict items. May be empty if no eligible items exist.

### Constraints

- The total `virtual_cost` of returned items MUST NOT exceed `max_block_cost`.
- The total spend count MUST NOT exceed `config.max_spends_per_block`.
- No two returned items may spend the same coin (conflict-free).
- Parents MUST appear before children in the returned vector (topological order).
- The function acquires a read lock on `pool_lock`; it does not mutate state.

## Acceptance Criteria

- [ ] The function `select_for_block` is publicly exported and has the exact signature specified above.
- [ ] The function accepts `max_block_cost`, `height`, and `timestamp` parameters.
- [ ] The return type is `Vec<Arc<MempoolItem>>`.
- [ ] Only items from the active pool are returned (no pending, no conflict cache items).
- [ ] The total cost of returned items does not exceed `max_block_cost`.
- [ ] The total spend count does not exceed `config.max_spends_per_block`.
- [ ] No two returned items conflict (share a spent coin).
- [ ] Items are returned in topological order (parents before children).

---

## Implementation Notes

- **Primary file:** `src/selection/mod.rs` or `src/selection/select.rs`
- **Public export:** `src/lib.rs` must re-export via `Mempool::select_for_block`.
- **Locking:** Acquires `pool_lock` as a read lock. Does not hold `pending_lock` or `conflict_lock`.
- **Companion methods:** `select_for_block_with_costs()` returns `Vec<BundleSelection>` with cost/fee metadata. `select_for_block_with_strategy()` allows caller-provided `BlockSelectionStrategy`.
- **Caller workflow:** `on_new_block()` -> resubmit retries -> `select_for_block()`.

| Signature compiles | Unit test | Call `select_for_block` with valid types | Compiles without error |

| Topological order | Unit test | Add parent + child CPFP items; call select | Parent appears before child |
