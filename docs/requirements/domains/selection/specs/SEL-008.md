## Summary

After the best-selection comparator ([SEL-007](SEL-007.md)) chooses the winning candidate set, the items are re-ordered into their final output order. The final order is topological (parents before children in CPFP dependency chains), with items at the same topological depth sorted by fee-per-virtual-cost descending. All sort orders include `height_added` and `spend_bundle_id` as deterministic tiebreakers.

This ordering is critical for the block producer. CPFP child transactions reference coins created by their parents, so parents must appear first in the block. Within each topological layer, high-density transactions appear first to maximize fee efficiency if the block is later truncated.

Chia's output ordering relies on SQL `ORDER BY priority DESC, seq ASC` ([`mempool.py:605`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool.py#L605)) which does not account for CPFP dependency ordering. dig-mempool explicitly enforces topological order to support CPFP dependency chains.

## Specification

### Ordering Rules

The final output `Vec<Arc<MempoolItem>>` is ordered as follows:

1. **Topological order:** Items are grouped into topological layers based on their CPFP dependency graph. An item is in layer 0 if it has no dependencies (`depends_on` is empty). An item is in layer N if the maximum layer of any of its parents is N-1.

2. **Within each layer:** Items are sorted by:
   | Priority | Field | Direction |
   |----------|-------|-----------|
   | 1 | `fee_per_virtual_cost_scaled` | Descending |
   | 2 | `height_added` | Ascending |
   | 3 | `spend_bundle_id` | Ascending |

3. **Layer 0 items** appear first, then layer 1, then layer 2, etc.

### Singleton Chain Ordering

Singleton chains (multiple items spending sequential versions of the same singleton, tracked via `singleton_spends`) are treated as dependency chains during final ordering. Items within a singleton chain are ordered by lineage (oldest version first), ensuring the block producer can rebase lineage proofs in sequence.

### Determinism

All sort orders include `height_added` ascending and `spend_bundle_id` ascending as final tiebreakers. Since `height_added` is set at admission time and `spend_bundle_id` is a deterministic hash, the output is fully deterministic given the same input set.

### Formal Algorithm

```
1. Compute topological layer for each item:
   layer(item) = 0                           if item.depends_on is empty
   layer(item) = 1 + max(layer(p) for p in item.depends_on)  otherwise

2. Group items by layer.

3. Within each layer, sort by:
   (fee_per_virtual_cost_scaled DESC, height_added ASC, spend_bundle_id ASC)

4. Concatenate layers in ascending order: layer 0 ++ layer 1 ++ ... ++ layer N.
```

## Acceptance Criteria

- [ ] Parents appear before children in the output vector.
- [ ] Items at the same topological depth are sorted by FPC descending.
- [ ] `height_added` ascending is used as a tiebreaker.
- [ ] `spend_bundle_id` ascending is used as a final tiebreaker.
- [ ] Singleton chain items are ordered by lineage (oldest version first).
- [ ] The output is deterministic given the same input set.
- [ ] Items with no dependencies (layer 0) appear first.

---

## Implementation Notes

- **Primary file:** `src/selection/order.rs` or `src/selection/mod.rs`
- **Topological sort:** Use Kahn's algorithm or a simple depth-first layer assignment. With max dependency depth of 25, the graph is shallow.
- **Singleton chain interleaving:** If a singleton chain spans multiple topological layers (e.g., layer 0 singleton -> layer 1 non-singleton -> layer 2 singleton update), each item appears in its correct layer. The block producer handles lineage rebasing.
- **Performance:** Topological sort on a set of up to 6000 items with max depth 25 is O(n) time.

| Parent before child | Unit test | Parent P, child C depends on P | P appears before C in output |
| Same-layer FPC ordering | Unit test | Layer 0: A (FPC=100), B (FPC=50) | A before B |
