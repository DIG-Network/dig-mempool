## Summary

`on_new_block()` is the primary lifecycle method called when a new block is confirmed on the L2 chain. It performs the critical state transitions that keep the mempool consistent with the current chain tip: removing confirmed transactions, expiring stale ones, collecting items eligible for retry, and updating the fee estimator.

This corresponds to Chia's `MempoolManager.new_peak()` ([`mempool_manager.py:858-937`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/mempool_manager.py#L858)) and `Mempool.new_tx_block()` ([`mempool.py:329-345`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/mempool.py#L329)). On Chia, `new_peak()` re-validates drained items inline under the lock. dig-mempool returns them as `RetryBundles` for the caller to resubmit with fresh coin records, maintaining the "no I/O" principle.

## Specification

### Function Signature

```rust
impl Mempool {
    pub fn on_new_block(
        &self,
        height: u64,
        timestamp: u64,
        spent_coin_ids: &[Bytes32],
        confirmed_bundles: &[ConfirmedBundleInfo],
    ) -> RetryBundles;
}
```

### Processing Steps (in order)

1. **Remove confirmed items + cascade**: For each coin ID in `spent_coin_ids`, look up the spending bundle in `coin_index`. Remove that bundle from the active pool. Cascade-evict all dependents (depth-first via `dependents` graph). Fire `on_item_removed(Confirmed)` for directly confirmed items and `on_item_removed(CascadeEvicted)` for their dependents.

2. **Remove expired items + cascade**: Query active items where `assert_before_height <= height` or `assert_before_seconds <= timestamp`. Remove each expired item and cascade-evict dependents. Fire `on_item_removed(Expired)`. Chia: `new_tx_block()` SQL query at [`mempool.py:336-340`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/mempool.py#L336).

3. **Collect pending promotions**: Extract pending items whose `assert_height <= height`. These are timelocked items that are now satisfiable. Return as `pending_promotions`. Chia: `_pending_cache.drain(new_peak.height)` ([`pending_tx_cache.py:91-108`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L91)).

4. **Collect conflict retries**: Extract conflict cache items whose conflicting active item was removed in step 1 or 2. Return as `conflict_retries`. Chia: `_conflict_cache.drain()` ([`pending_tx_cache.py:40-44`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L40)).

5. **Update fee estimator**: Feed `confirmed_bundles` to the `FeeTracker` via `record_confirmed_block()`. See [FEE-004](../../fee_estimation/specs/FEE-004.md).

6. **Update tracking**: Set internal `height` and `timestamp` to the new values.

7. **Return `RetryBundles`**: Contains `conflict_retries`, `pending_promotions`, and `cascade_evicted` bundle IDs.

### Locking

`on_new_block()` acquires all write locks briefly (SPEC Section 15). This is infrequent (once per block) and acceptable.

## Acceptance Criteria

- [ ] `on_new_block()` is publicly exported on the `Mempool` struct.
- [ ] Confirmed items (matching `spent_coin_ids`) are removed from the active pool.
- [ ] Dependents of confirmed items are cascade-evicted.
- [ ] Expired items (past `assert_before_height` or `assert_before_seconds`) are removed.
- [ ] Dependents of expired items are cascade-evicted.
- [ ] Pending items with satisfied timelocks are collected in `pending_promotions`.
- [ ] Conflict cache items whose conflict is resolved are collected in `conflict_retries`.
- [ ] Fee estimator is updated with `confirmed_bundles` data.
- [ ] Internal height and timestamp are updated.
- [ ] Event hooks are fired with correct `RemovalReason` for each removal.

---

## Implementation Notes

- **Primary file:** `src/lifecycle.rs` or `src/mempool.rs`
- **Cascade eviction order:** Depth-first from the removed parent. Each cascade-evicted child records its parent's bundle ID in `RemovalReason::CascadeEvicted { parent_id }`.
- **Conflict retry eligibility:** A conflict cache item is retryable if ALL of its conflicting active items were removed (confirmed, expired, or cascade-evicted).
- **Pending promotion:** The `SpendBundle` is extracted from the pending pool (removed from pending state). The caller must resubmit with current coin records.
- **Edge cases:**
  - `spent_coin_ids` contains coins not in the mempool: silently ignored.
  - Same coin confirmed by multiple IDs in `spent_coin_ids`: handled idempotently.
  - No items to remove: returns empty `RetryBundles`.
