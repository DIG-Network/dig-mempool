## Summary

`RemovalReason` is an enum that describes why a mempool item was removed. It is passed to `MempoolEventHook::on_item_removed()` and used internally for bookkeeping, logging, and metrics.

Chia's equivalent is `MempoolRemoveReason` ([`mempool.py:87-91`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/mempool.py#L87)) with four variants: `CONFLICT`, `BLOCK_INCLUSION`, `POOL_FULL`, `EXPIRED`. dig-mempool extends this with `ReplacedByFee`, `CascadeEvicted`, `ExplicitRemoval`, and `Cleared` to provide finer-grained removal tracking, particularly for CPFP cascade eviction and reorg recovery.

The enum derives `Debug + Clone + PartialEq` for testability (SPEC Section 1.3, Design Decision #14).

## Specification

### Enum Definition

```rust
#[derive(Debug, Clone, PartialEq)]
pub enum RemovalReason {
    /// Included in a confirmed block.
    Confirmed,
    /// Replaced by a higher-fee bundle (RBF).
    ReplacedByFee { replacement_id: Bytes32 },
    /// Parent was removed; this item's inputs no longer exist.
    CascadeEvicted { parent_id: Bytes32 },
    /// Expired (assert_before_height or assert_before_seconds passed).
    Expired,
    /// Evicted due to capacity pressure (low fee-per-cost).
    CapacityEviction,
    /// Explicitly removed by caller via remove().
    ExplicitRemoval,
    /// Mempool cleared (reorg recovery).
    Cleared,
}
```

### Variant Usage

| Variant | Used By | Notes |
|---------|---------|-------|
| `Confirmed` | `on_new_block()` step 1 | Item's coins were in `spent_coin_ids` |
| `ReplacedByFee` | `submit()` Phase 2 RBF | Contains the replacing bundle's ID |
| `CascadeEvicted` | Cascade eviction (various) | Contains the removed parent's bundle ID |
| `Expired` | `on_new_block()` step 2 | `assert_before_height` or `assert_before_seconds` passed |
| `CapacityEviction` | `evict_lowest_percent()`, capacity mgmt | Lowest `descendant_score` evicted |
| `ExplicitRemoval` | `remove()` public API | Caller-initiated removal |
| `Cleared` | `clear()` | Reorg recovery reset |

### Derives

- `Debug`: Required for error messages and logging.
- `Clone`: Required because `RemovalReason` may be passed to multiple hooks.
- `PartialEq`: Required for test assertions.

## Acceptance Criteria

- [ ] `RemovalReason` is publicly exported from the crate.
- [ ] All seven variants are present: `Confirmed`, `ReplacedByFee`, `CascadeEvicted`, `Expired`, `CapacityEviction`, `ExplicitRemoval`, `Cleared`.
- [ ] `ReplacedByFee` contains `replacement_id: Bytes32`.
- [ ] `CascadeEvicted` contains `parent_id: Bytes32`.
- [ ] Derives `Debug + Clone + PartialEq`.
- [ ] Each variant is used in the correct context as documented.
- [ ] Passed to `MempoolEventHook::on_item_removed()`.

---

## Implementation Notes

- **Primary file:** `src/types.rs` or `src/hooks.rs`
- **No `Copy`:** The `Bytes32` fields prevent `Copy` derive, which is acceptable.
- **Display:** Consider implementing `Display` for user-facing logging, though `Debug` is sufficient for most uses.
- **Extensibility:** New variants may be added in future versions (e.g., `PolicyEviction`). Consumers should handle unknown variants gracefully if matching exhaustively.
