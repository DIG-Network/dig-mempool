## Summary

`MempoolEventHook` is a trait that enables synchronous callbacks on mempool state mutations. It allows external systems (logging, metrics, notifications, wallet updates) to observe mempool changes without polling.

Hooks are called synchronously under the write lock, which means implementations MUST be fast and non-blocking. All callback methods have default no-op implementations, so callers only override the events they care about.

This is a dig-mempool extension not present in Chia L1, which has no event hook mechanism (SPEC Section 14.2). Multiple hooks can be registered via `add_event_hook()`, and they are all called for each event.

---

## Specification

### Trait Definition

```rust
pub trait MempoolEventHook: Send + Sync {
    fn on_item_added(&self, _item: &MempoolItem) {}
    fn on_item_removed(&self, _bundle_id: &Bytes32, _reason: RemovalReason) {}
    fn on_block_selected(&self, _items: &[Arc<MempoolItem>]) {}
    fn on_conflict_cached(&self, _bundle_id: &Bytes32) {}
    fn on_pending_added(&self, _item: &MempoolItem) {}
}
```

### Registration

```rust
impl Mempool {
    pub fn add_event_hook(&self, hook: Arc<dyn MempoolEventHook>);
}
```

### Callback Semantics

| Callback | Triggered When | Arguments |
|----------|----------------|-----------|
| `on_item_added` | Item inserted into active pool | Reference to the new `MempoolItem` |
| `on_item_removed` | Item removed from active pool | Bundle ID and `RemovalReason` |
| `on_block_selected` | `select_for_block()` returns | Slice of selected items |
| `on_conflict_cached` | Bundle added to conflict cache | Bundle ID |
| `on_pending_added` | Item added to pending pool | Reference to the pending `MempoolItem` |

### Constraints

- **Send + Sync:** Required because hooks are called from write-lock context which may be accessed from multiple threads.
- **Non-blocking:** Implementations MUST NOT perform I/O, acquire external locks, or block. Violations risk deadlock or degraded throughput.
- **Default implementations:** All methods have empty defaults. Implementors override only needed methods.
- **Multiple hooks:** `add_event_hook()` can be called multiple times. All registered hooks are called for each event.
- **Call order:** Hooks are called in registration order.
- **Under write lock:** Hooks are called while the mempool holds the write lock. This ensures consistency (the hook sees the state as it exists at the moment of the event) but requires fast execution.

## Acceptance Criteria

- [ ] `MempoolEventHook` trait is publicly exported.
- [ ] Trait requires `Send + Sync`.
- [ ] All five callback methods are defined with default no-op implementations.
- [ ] `add_event_hook()` accepts `Arc<dyn MempoolEventHook>`.
- [ ] Multiple hooks can be registered.
- [ ] `on_item_added` is called on active pool insertion.
- [ ] `on_item_removed` is called on active pool removal with correct `RemovalReason`.
- [ ] `on_block_selected` is called by `select_for_block()`.
- [ ] `on_conflict_cached` is called when a bundle enters the conflict cache.
- [ ] `on_pending_added` is called when an item enters the pending pool.

## Implementation Notes

- **Primary file:** `src/hooks.rs` or `src/traits.rs`
- **Storage:** `hooks: Vec<Arc<dyn MempoolEventHook>>` protected by `hooks_lock: RwLock` (SPEC Section 8.8-8.9).
- **Hook invocation:** Read `hooks_lock` under the parent write lock. Iterate and call each hook.
- **Panic safety:** Consider catching panics from hook implementations to prevent one misbehaving hook from corrupting mempool state.
- **Edge cases:**
  - No hooks registered: no-op (no iteration).
  - Hook panics: should not propagate (catch and log).
