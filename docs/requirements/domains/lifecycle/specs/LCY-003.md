## Summary

The mempool enforces a specific caller workflow for block confirmation and item recycling. After a new block is confirmed, the caller MUST follow the sequence: `on_new_block()` -> `submit()` retries -> `select_for_block()`. This ensures that promoted pending items and conflict retries are re-validated with current chain state before being eligible for block selection.

This matches Chia L1's sequencing where `new_peak()` drains caches and re-validates inline ([`mempool_manager.py:858-937`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool_manager.py#L858)). dig-mempool externalizes the re-validation step to maintain the "no I/O" principle: the mempool does not fetch coin records itself.

Promoted items are NOT automatically eligible for block selection. They must be resubmitted via `submit()` so that the mempool can re-validate them against current coin state. A pending item's coins may have been spent between the time it was submitted and the time its timelock becomes satisfiable.

## Specification

### Required Workflow

```
1. on_new_block(height, timestamp, spent_coin_ids, confirmed_bundles) -> RetryBundles
2. For each bundle in RetryBundles.conflict_retries:
      submit(bundle, current_coin_records, height, timestamp)
3. For each bundle in RetryBundles.pending_promotions:
      submit(bundle, current_coin_records, height, timestamp)
4. select_for_block(max_block_cost, height, timestamp) -> Vec<Arc<MempoolItem>>
```

### Rules

- **Promoted items require resubmission:** Items returned in `pending_promotions` and `conflict_retries` are raw `SpendBundle`s, not `MempoolItem`s. They must go through the full admission pipeline (CLVM validation, conflict detection, etc.) again.
- **Fresh coin records:** The caller MUST provide current `coin_records` for the resubmission. Coins may have been spent or created since the item was originally submitted.
- **Failure is expected:** Some retried bundles may fail resubmission (coins spent, new conflicts). This is normal and expected.
- **Selection after retries:** `select_for_block()` should be called after all retries are processed, so newly re-admitted items are eligible for selection.

### Why Not Automatic Promotion

The mempool cannot automatically promote items because:
1. It does not have access to current coin records (no I/O).
2. Coin state may have changed since original submission.
3. CLVM validation must be re-run to ensure the bundle is still valid.
4. New conflicts may exist with items admitted during the same block cycle.

---

## Acceptance Criteria

- [ ] Items in `pending_promotions` are not in the active pool after `on_new_block()`.
- [ ] Items in `conflict_retries` are not in the conflict cache after `on_new_block()`.
- [ ] Promoted items are only eligible for `select_for_block()` after successful `submit()`.
- [ ] Failed resubmissions do not corrupt mempool state.
- [ ] The workflow is documented in public API documentation.

---

## Implementation Notes

- **No enforcement mechanism:** The mempool does not enforce the workflow order at runtime. It is the caller's responsibility to follow the sequence. However, calling `select_for_block()` before resubmitting retries will simply not include those items.
- **Batch resubmission:** The caller MAY use `submit_batch()` for efficiency when resubmitting multiple retries.
- **Error handling:** The caller SHOULD handle resubmission failures gracefully (log and continue, do not retry indefinitely).

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| Promoted items not auto-eligible | Integration test | Submit timelocked item, advance past timelock, select_for_block without resubmit | Promoted item NOT in selection |
| Resubmitted items eligible | Integration test | Promote, resubmit, then select_for_block | Resubmitted item in selection |
| Failed resubmission harmless | Integration test | Promote, resubmit with stale coin records | MempoolError returned, pool unaffected |
| Conflict retry workflow | Integration test | Create conflict, confirm original, resubmit conflict | Previously conflicting bundle admitted |
| Batch resubmission works | Integration test | Collect retries, submit_batch all at once | All valid retries admitted |
| Select before resubmit is safe | Unit test | Call select_for_block before resubmitting | Returns items without promoted ones, no error |
