## Summary

After `dig_clvm::validate_spend_bundle()` returns a successful `SpendResult`, the mempool extracts the fee and validates it against the bundle's declared `RESERVE_FEE` condition. The fee is already computed by chia-consensus as `removal_amount - addition_amount` (the implicit fee from the difference between inputs and outputs). The `RESERVE_FEE` condition is a minimum fee guarantee declared by the bundle's puzzles -- if the actual fee is lower than the reserved amount, the bundle must be rejected.

On Chia L1, this check is performed in `validate_spend_bundle()` at [`mempool_manager.py:728`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool_manager.py#L728) where `fees < reserved_fee` triggers `Err.RESERVE_FEE_CONDITION_NOT_MET`. The fee itself comes from `OwnedSpendBundleConditions` which chia-consensus computes during CLVM execution. The mempool does not recompute the fee; it reads the pre-computed value.

The `RESERVE_FEE` condition (opcode 52) allows puzzle authors to enforce a minimum fee. This is important for security: a puzzle might require a minimum fee to prevent economic attacks. The chia-consensus CLVM runner sums all `RESERVE_FEE` conditions across all spends in the bundle into `conditions.reserve_fee`. The mempool's job is simply to enforce `fee >= reserve_fee`.

## Specification

### Fee Extraction

```rust
let fee: u64 = spend_result.fee;
// Equivalent to: spend_result.conditions.removal_amount - spend_result.conditions.addition_amount
// Already computed by chia-consensus in OwnedSpendBundleConditions
```

The fee is the implicit fee from the difference between total input value and total output value:
- `removal_amount` = sum of all spent coin amounts (u128, cast to u64 after conservation check)
- `addition_amount` = sum of all `CREATE_COIN` amounts (u128, cast to u64)
- `fee = removal_amount - addition_amount` (always non-negative after conservation check)

### RESERVE_FEE Check

```rust
let reserve_fee: u64 = spend_result.conditions.reserve_fee;
if fee < reserve_fee {
    return Err(MempoolError::InsufficientFee {
        required: reserve_fee,
        available: fee,
    });
}
```

`reserve_fee` is the sum of all `RESERVE_FEE` (opcode 52) conditions across all spends in the bundle. chia-consensus pre-sums this value during condition parsing. If no `RESERVE_FEE` conditions exist, `reserve_fee = 0` and this check always passes.

### Additional Extractions

After the RESERVE_FEE check, the following fields are also extracted from `SpendResult` for use in subsequent pipeline steps:

| Field | Source | Usage |
|-------|--------|-------|
| `fee` | `spend_result.fee` | Fee ordering, RBF comparison, virtual cost |
| `cost` | `spend_result.conditions.cost` | Cost limits, virtual cost, capacity management |
| `additions` | `spend_result.additions` | CPFP coin tracking, state deltas |
| `removals` | `spend_result.removals` | Conflict detection, state deltas |
| `conditions` | `spend_result.conditions` | Timelocks, flags, per-spend data |
| `num_spends` | `bundle.coin_spends.len()` | Virtual cost, spend count limits |

## Acceptance Criteria

- [ ] Fee is read from `SpendResult.fee`, not recomputed.
- [ ] `reserve_fee` is read from `SpendResult.conditions.reserve_fee`.
- [ ] If `fee < reserve_fee`, returns `MempoolError::InsufficientFee { required, available }`.
- [ ] If `fee >= reserve_fee`, the check passes and the pipeline continues.
- [ ] Zero-fee bundles with zero `reserve_fee` pass the check (both are 0).
- [ ] Multiple `RESERVE_FEE` conditions are summed correctly (by chia-consensus, not the mempool).
- [ ] The fee value stored in the resulting `MempoolItem` matches `SpendResult.fee`.

## Implementation Notes

- **Primary file:** `src/admission.rs` (fee extraction step)
- **Dependencies:** ADM-002 (produces the SpendResult), ADM-005 (uses fee for virtual cost)
- **Edge cases:**
  - `reserve_fee = 0` (no RESERVE_FEE conditions): check always passes. Most bundles.
  - `fee = 0` with `reserve_fee = 0`: valid (zero-fee transaction allowed when mempool is under 80% utilization).
  - `fee = reserve_fee`: exact match passes (not strict inequality).
  - Overflow: `removal_amount` and `addition_amount` are u128 in chia-consensus; the fee cast to u64 is safe after the conservation check ensures non-negative.
