> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 5.7

## Summary

The mempool reads two flags from the validated CLVM output to determine whether a bundle is eligible for identical-spend deduplication and singleton fast-forward optimization. These flags are NOT computed by the mempool -- they are set by chia-consensus's `MempoolVisitor` during CLVM execution when `MEMPOOL_MODE` is active. The mempool's role is strictly to read these pre-computed flags from `OwnedSpendConditions.flags` and store them in the `MempoolItem`.

The `ELIGIBLE_FOR_DEDUP` flag (0x1) indicates that a coin spend uses canonical encoding, meaning identical spends across different bundles can be deduplicated at the block generator level (sharing a single puzzle/solution copy). This is set by chia-consensus when the spend passes canonical form checks. Chia L1 reads this flag at [`mempool_manager.py:662-663`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool_manager.py#L662).

The `ELIGIBLE_FOR_FF` flag (0x4) indicates that a coin spend is structurally eligible for singleton fast-forward. This is set by chia-consensus when the puzzle structure supports fast-forward (singleton top layer with compatible inner puzzle). The mempool reads this flag; the caller is responsible for having called `supports_fast_forward()` and `SingletonLayer::parse_puzzle()` before submission to provide the `SingletonLineageInfo`.

## Specification

### Flag Constants

```rust
// From chia_consensus::flags (or chia_consensus::gen::flags)
const ELIGIBLE_FOR_DEDUP: u32 = 0x1;
const ELIGIBLE_FOR_FF: u32 = 0x4;
```

### ELIGIBLE_FOR_DEDUP Extraction

```rust
let eligible_for_dedup = if config.enable_identical_spend_dedup {
    spend_result.conditions.spends.iter().all(|s| {
        s.flags & ELIGIBLE_FOR_DEDUP != 0
    })
} else {
    false
};
```

A bundle is `eligible_for_dedup = true` only if ALL spends in the bundle have the `ELIGIBLE_FOR_DEDUP` flag set. If any spend lacks the flag, the entire bundle is not eligible for dedup.

### ELIGIBLE_FOR_FF Extraction

```rust
let has_ff_eligible_spend = if config.enable_singleton_ff {
    spend_result.conditions.spends.iter().any(|s| {
        s.flags & ELIGIBLE_FOR_FF != 0
    })
} else {
    false
};
```

If any spend has `ELIGIBLE_FOR_FF`, the bundle may contain a singleton that can be fast-forwarded. The caller provides `SingletonLineageInfo` at submission time (extracted via `SingletonLayer::parse_puzzle()` before calling `submit()`). The mempool does NOT parse puzzles.

### What the Mempool Does NOT Do

The mempool MUST NOT:
- Run CLVM to determine flag eligibility (chia-consensus does this)
- Check canonical encoding (chia-consensus does this via MempoolVisitor)
- Parse puzzle structures for singleton detection (caller does this)
- Call `supports_fast_forward()` (caller does this before submission)

### Storage in MempoolItem

```rust
MempoolItem {
    eligible_for_dedup,
    singleton_lineage: /* provided by caller or None */,
    // ...
}
```

## Acceptance Criteria

- [ ] `ELIGIBLE_FOR_DEDUP` (0x1) is read from `OwnedSpendConditions.flags` for each spend.
- [ ] `eligible_for_dedup` is `true` only if ALL spends have the flag set.
- [ ] `ELIGIBLE_FOR_FF` (0x4) is read from `OwnedSpendConditions.flags` for each spend.
- [ ] Flag reading respects `config.enable_identical_spend_dedup` and `config.enable_singleton_ff`.
- [ ] The mempool does NOT compute these flags itself (no CLVM, no canonical checks, no puzzle parsing).
- [ ] Flags are stored correctly in the resulting `MempoolItem`.
- [ ] Bundles with no spends having either flag set have `eligible_for_dedup = false` and no singleton lineage.

## Implementation Notes

- **Primary file:** `src/admission.rs` (flag extraction step, Phase 2)
- **Dependencies:** ADM-002 (provides SpendResult with conditions containing flags)
- **Edge cases:**
  - Bundle with single spend: `eligible_for_dedup` = that spend's flag value.
  - Bundle with mixed flags (some dedup-eligible, some not): `eligible_for_dedup = false` (ALL must have it).
  - `config.enable_identical_spend_dedup = false`: `eligible_for_dedup` is always `false` regardless of flags.
  - `config.enable_singleton_ff = false`: FF handling is skipped entirely.
  - Multiple FF-eligible spends in one bundle: unusual but possible; only one `singleton_lineage` can be tracked per item.
