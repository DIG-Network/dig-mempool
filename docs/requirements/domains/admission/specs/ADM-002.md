## Summary

The mempool validates every submitted `SpendBundle` by calling `dig_clvm::validate_spend_bundle()`. This is the expensive computational step of the admission pipeline -- it performs a full CLVM dry-run (executing puzzles with solutions), verifies the BLS aggregate signature, parses all conditions, enforces cost limits, and checks conservation (inputs >= outputs + fee). Only bundles that pass this validation are considered for admission.

This design matches Chia L1's approach where `MempoolManager.validate_spend_bundle()` ([`mempool_manager.py:609-833`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool_manager.py#L609)) calls `validate_clvm_and_signature()` ([`mempool_manager.py:445`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool_manager.py#L445)) which delegates to `get_name_puzzle_conditions()` for CLVM execution and condition extraction. The key difference is that dig-mempool delegates all of this to `dig-clvm` as a single function call, keeping the mempool free of CLVM execution logic.

The validation runs in Phase 1 (lock-free), meaning it does NOT hold the mempool write lock. This is critical for performance: CLVM execution can take tens of milliseconds per bundle, and running it under lock would serialize all submissions. By validating lock-free, multiple `submit()` calls can run CLVM concurrently, with only the fast Phase 2 state mutation requiring serialization.

## Specification

### Validation Call

```rust
// Phase 1: Lock-free CLVM validation
let bundle_id = bundle.name();
let ctx = ValidationContext {
    height: current_height as u32,
    timestamp: current_timestamp,
    constants: self.constants.clone(),
    coin_records: coin_records.clone(),
    ephemeral_coins: HashSet::new(),  // populated for CPFP candidates
};
let config = ValidationConfig {
    flags: MEMPOOL_MODE,  // strict validation
    ..Default::default()
};
let spend_result = dig_clvm::validate_spend_bundle(
    &bundle, &ctx, &config, Some(&mut bls_cache)
)?;
```

### What dig-clvm Validates

| Check | dig-clvm Location | Failure Error |
|-------|-------------------|---------------|
| Duplicate spend detection | `validate.rs:39-48` | `ValidationError::DoubleSpend` |
| Coin existence | `validate.rs:50-62` | `ValidationError::CoinNotFound` |
| Already-spent coins | `validate.rs:54-56` | `ValidationError::AlreadySpent` |
| Puzzle hash match | Inside `run_spendbundle()` | `ValidationError::Clvm` |
| CLVM execution | `validate.rs:79-102` via `run_spendbundle()` | `ValidationError::Clvm` |
| BLS signature | `validate.rs:107-113` via `BlsCache::aggregate_verify()` | `ValidationError::SignatureFailed` |
| Cost limits | `validate.rs:126-131` | `ValidationError::CostExceeded` |
| Conservation | `validate.rs:148-159` | `ValidationError::ConservationViolation` |

### Error Conversion

`dig_clvm::ValidationError` is converted to `MempoolError::ValidationError(e.to_string())` for the mempool's `Clone + PartialEq` derive support. The original error information is preserved in the string representation.

### BLS Cache

The mempool maintains an internal `BlsCache` (from `chia-bls`) that is passed to `validate_spend_bundle()`. This cache stores verified BLS pairings and reuses them across submissions, avoiding redundant signature verification for bundles that share signers. The cache is thread-safe behind the same `RwLock` or a separate mutex.

### MEMPOOL_MODE Flag

Validation MUST use `MEMPOOL_MODE` (from `chia_consensus::flags`), which enables stricter rules:
- Reject unknown condition opcodes
- Stricter cost accounting
- Canonical encoding checks for dedup eligibility

This matches Chia L1's mempool validation at [`mempool_manager.py:700-701`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool_manager.py#L700).

## Acceptance Criteria

- [ ] `submit()` calls `dig_clvm::validate_spend_bundle()` on every submission.
- [ ] The `ValidationContext` is constructed with the caller-provided `coin_records`, `current_height`, and `current_timestamp`.
- [ ] The `ValidationConfig` uses `MEMPOOL_MODE` flags.
- [ ] An internal `BlsCache` is passed to `validate_spend_bundle()` for pairing reuse.
- [ ] Invalid bundles (bad CLVM, bad signature, conservation violation) are rejected before any mempool state changes.
- [ ] `dig_clvm::ValidationError` is converted to `MempoolError::ValidationError(String)`.
- [ ] CLVM validation runs without holding the mempool write lock (Phase 1, lock-free).

## Implementation Notes

- **Primary file:** `src/admission.rs` (or inline in `src/mempool.rs`)
- **Dependencies:** Depends on `dig-clvm` crate. References ADM-001 (entry point that orchestrates this call).
- **Edge cases:**
  - `BlsCache` contention: If multiple threads share the same cache via mutex, ensure the lock is released before Phase 2 begins.
  - `coin_records` mismatch: If the caller provides stale coin records (coin already spent on-chain but not marked as spent), dig-clvm will reject with `AlreadySpent`.
  - Large bundles with many spends: CLVM cost may exceed limits, rejected by dig-clvm's cost enforcement.
