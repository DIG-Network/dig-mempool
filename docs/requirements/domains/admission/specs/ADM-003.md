# ADM-003: Dedup Check via Seen-Cache Before CLVM Validation

> **Authoritative requirement:** [ADM-003](../NORMATIVE.md#ADM-003)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 5.2

## Summary

Before performing expensive CLVM validation, the mempool checks whether it has already seen a bundle with the same ID. The bundle ID is computed via `SpendBundle::name()` (from `chia-protocol`), which produces the canonical `sha256` hash of the bundle's Streamable serialization. This ID is checked against the seen-cache, active pool items, pending pool items, and conflict cache. If found in any of these, the submission is immediately rejected with `MempoolError::AlreadySeen` without running CLVM.

This is a critical DoS protection mechanism. Without it, an attacker could repeatedly submit the same invalid bundle, forcing the mempool to re-run CLVM validation (which takes 10-50ms per bundle) on every submission. By caching seen bundle IDs, the mempool can reject duplicates in O(1) time with a simple hash lookup.

The seen-cache is populated **immediately** after computing the bundle ID, before CLVM validation begins. This matches Chia L1's approach where `MempoolManager.seen_bundle_hashes` ([`mempool_manager.py:298`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/mempool_manager.py#L298)) is populated before validation in `add_spend_bundle()` ([`mempool_manager.py:553-558`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/mempool_manager.py#L553)). Note that this means even failed submissions are cached -- an invalid bundle will be rejected quickly on retry rather than re-validated.

## Specification

### Bundle ID Computation

```rust
let bundle_id: Bytes32 = bundle.name();
```

`SpendBundle::name()` is defined in `chia-protocol` and computes the canonical hash using Chia's `Streamable` serialization. No custom serialization is needed. This matches Chia's `SpendBundle.name()` ([`coin.py:43`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/types/coin.py#L43)).

### Dedup Check Order

```
1. Check seen_cache          -> AlreadySeen
2. Check active_items        -> AlreadySeen
3. Check pending_items       -> AlreadySeen
4. Check conflict_cache      -> AlreadySeen
5. Add to seen_cache         (immediately, before CLVM)
6. Proceed to CLVM validation (ADM-002)
```

### Seen-Cache Properties

| Property | Value |
|----------|-------|
| Data structure | `HashSet<Bytes32>` or LRU cache |
| Capacity | `config.max_seen_cache_size` (default: 10,000) |
| Eviction | LRU when capacity exceeded |
| Thread safety | Behind `RwLock` (read lock for check, write lock for insert) |
| Persistence | NOT persisted across restarts (ephemeral, DoS protection only) |
| Cleared on | `clear()` (reorg recovery), capacity eviction |

### Phase 1 Race Condition

In the concurrent submission model, two identical bundles can pass the seen-cache check simultaneously (both see "not found" before either inserts). This is handled by a re-check in Phase 2 (under write lock) before insertion. The seen-cache check in Phase 1 is an optimization, not a correctness guarantee.

## Acceptance Criteria

- [ ] Bundle ID is computed via `SpendBundle::name()` from `chia-protocol`.
- [ ] The seen-cache, active items, pending items, and conflict cache are all checked before CLVM validation.
- [ ] Duplicate bundles return `MempoolError::AlreadySeen(bundle_id)`.
- [ ] The bundle ID is added to the seen-cache immediately after the check, before CLVM validation runs.
- [ ] The seen-cache respects `config.max_seen_cache_size` with LRU eviction.
- [ ] Even invalid bundles are cached in the seen-cache (prevents repeated CLVM validation of bad bundles).
- [ ] Phase 2 includes a re-check for duplicates that passed Phase 1 concurrently.

---

## Implementation Notes

- **Primary file:** `src/admission.rs` (dedup check step)
- **Dependencies:** ADM-001 (entry point), ADM-002 (CLVM validation that this protects)
- **Edge cases:**
  - Seen-cache full: LRU eviction makes the oldest entries re-submittable. This is acceptable because the active/pending/conflict checks still catch admitted bundles.
  - Race condition: Two threads submit identical bundles simultaneously. Both pass Phase 1 dedup. Phase 2 re-check catches the second one.
  - `clear()` resets the seen-cache, allowing previously-seen bundles to be resubmitted (correct behavior after reorg).

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| `test_duplicate_rejected` | Simulator | Submit the same valid bundle twice | Second returns `Err(MempoolError::AlreadySeen(_))` |
| `test_invalid_bundle_cached` | Simulator | Submit an invalid bundle, then submit it again | Second returns `AlreadySeen` (not re-validated) |
| `test_seen_cache_lru_eviction` | Unit | Fill seen-cache to capacity, submit one more, then retry the oldest evicted | Oldest can be re-submitted (evicted from seen-cache) |
| `test_dedup_before_clvm` | Unit | Verify that dedup check runs before CLVM by timing or mock | No CLVM invocation for duplicate |
| `test_bundle_id_matches_chia` | Unit | Compute `SpendBundle::name()` and verify it matches expected hash | Hash matches Chia L1 computation |
| `test_concurrent_dedup_race` | Integration | Submit same bundle from two threads concurrently | At most one succeeds; the other gets AlreadySeen or duplicate rejection |
