# ADM-008: Batch Submission — Concurrent Validation, Sequential Insertion

> **Authoritative requirement:** [ADM-008](../NORMATIVE.md#ADM-008)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 3.2

## Summary

`submit_batch()` accepts a vector of `SpendBundle`s and processes them using a two-phase approach: Phase 1 (CLVM validation) runs concurrently across all bundles without holding the mempool write lock, and Phase 2 (state mutation) processes all validated bundles sequentially under a single write lock acquisition. This design maximizes throughput for bulk submissions (e.g., receiving a batch from a peer node) while maintaining correctness guarantees.

The batch approach improves on Chia L1's serial submission model where `add_spend_bundle()` is called one at a time ([`mempool_manager.py:538`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool_manager.py#L538)). The key innovation is that earlier entries in the batch can create coins that later entries depend on (CPFP), because Phase 2 processes them in order -- inserting item N before evaluating item N+1's dependencies.

Results are returned in the same order as inputs, with one `Result<SubmitResult, MempoolError>` per input bundle. A failure in one bundle does not prevent processing of subsequent bundles.

## Specification

### Function Signature

```rust
impl Mempool {
    pub fn submit_batch(
        &self,
        bundles: Vec<SpendBundle>,
        coin_records: &HashMap<Bytes32, CoinRecord>,
        current_height: u64,
        current_timestamp: u64,
    ) -> Vec<Result<SubmitResult, MempoolError>>;

    pub fn submit_batch_with_policy(
        &self,
        bundles: Vec<SpendBundle>,
        coin_records: &HashMap<Bytes32, CoinRecord>,
        current_height: u64,
        current_timestamp: u64,
        policy: &dyn AdmissionPolicy,
    ) -> Vec<Result<SubmitResult, MempoolError>>;
}
```

### Two-Phase Processing

```
Phase 1: Concurrent Validation (lock-free)
┌─────────────────────────────────────────────────────────┐
│  Bundle 0: dedup → CLVM → fee → cost → timelocks       │
│  Bundle 1: dedup → CLVM → fee → cost → timelocks       │  concurrent
│  Bundle 2: dedup → CLVM → fee → cost → timelocks       │
│  ...                                                     │
└─────────────────────────────────────────────────────────┘

Phase 2: Sequential Insertion (single write lock)
┌─────────────────────────────────────────────────────────┐
│  acquire write lock                                      │
│  Bundle 0: re-check dedup → flags → CPFP → conflict →  │
│            RBF → capacity → insert                       │  sequential
│  Bundle 1: re-check dedup → flags → CPFP → conflict →  │
│            RBF → capacity → insert                       │
│  Bundle 2: ...                                           │
│  release write lock                                      │
└─────────────────────────────────────────────────────────┘
```

### CPFP Within Batch

Because Phase 2 processes bundles sequentially in input order, bundle N's additions are visible to bundle N+1's CPFP dependency resolution. This enables a batch like:

```
bundles[0]: creates coin C
bundles[1]: spends coin C (CPFP child of bundles[0])
```

### Concurrent Phase 1

Phase 1 SHOULD use `rayon::par_iter()` or similar parallel iteration for CLVM validation. Each bundle's validation is independent (no shared state mutation). The BLS cache requires synchronization if shared.

### Error Isolation

Each bundle is processed independently. A failure in bundle N does not prevent processing of bundle N+1. However, if bundle N fails in Phase 2 and bundle N+1 depends on a coin created by bundle N, bundle N+1 will fail with `CoinNotFound` in its Phase 2 CPFP resolution.

### Result Ordering

Results MUST be returned in the same order as inputs:

```rust
assert_eq!(results.len(), bundles.len());
// results[i] corresponds to bundles[i]
```

## Acceptance Criteria

- [ ] `submit_batch()` accepts `Vec<SpendBundle>` and returns `Vec<Result<SubmitResult, MempoolError>>`.
- [ ] Phase 1 (CLVM validation) runs concurrently across all bundles, without the write lock.
- [ ] Phase 2 (state mutation) acquires the write lock once for all insertions.
- [ ] Bundles are processed sequentially in Phase 2, in input order.
- [ ] Earlier entries can create coins that later entries depend on (CPFP within batch).
- [ ] Results are returned in the same order as inputs.
- [ ] A failure in one bundle does not prevent processing of subsequent bundles.
- [ ] `submit_batch_with_policy()` applies the admission policy to each bundle in Phase 2.

## Implementation Notes

- **Primary file:** `src/mempool.rs` (batch entry point) and `src/admission.rs` (per-bundle logic)
- **Dependencies:** ADM-001 through ADM-007 (each bundle goes through the full admission pipeline)
- **Edge cases:**
  - Empty batch (`Vec::new()`): returns empty `Vec`.
  - Single-element batch: equivalent to `submit()` but with batch overhead.
  - All bundles in batch are duplicates of each other: first succeeds, rest get `AlreadySeen` in Phase 2 re-check.
  - Circular CPFP within batch (A depends on B, B depends on A): impossible since Phase 2 is sequential (A processed first, B cannot create coins A depends on before A is processed).
  - Large batch (1000+ bundles): Phase 1 parallelism is bounded by thread pool. Phase 2 holds write lock for longer.
