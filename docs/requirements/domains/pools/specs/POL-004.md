## Summary

The pending pool holds validated transactions that are future-timelocked -- their `assert_height` is greater than the current chain height, so they cannot be included in a block yet. These items wait in the pending pool until their timelocks are satisfied, at which point `on_new_block()` promotes them for re-submission to the active pool.

This corresponds to Chia's `PendingTxCache` ([`pending_tx_cache.py:50-111`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L50)), which stores timelocked transactions with count and cost limits.

The pending pool is separate from the active pool and uses its own lock (`pending_lock`) to minimize contention.

## Specification

### Data Structures

```rust
// Pending pool storage (protected by pending_lock: RwLock)
pending:             HashMap<Bytes32, Arc<MempoolItem>>  // bundle_id -> item
pending_coin_index:  HashMap<Bytes32, Bytes32>           // spent_coin_id -> bundle_id
pending_cost:        u64                                 // aggregate cost
```

### Capacity Limits

| Parameter | Default | Source |
|-----------|---------|--------|
| `max_pending_count` | 3,000 | `MempoolConfig.max_pending_count`. Chia: `PendingTxCache._cache_max_size = 3000` ([`pending_tx_cache.py:53`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L53)) |
| `max_pending_cost` | 1x block cost | `MempoolConfig.max_pending_cost`. Chia: `PendingTxCache._cache_max_total_cost` ([`pending_tx_cache.py:52`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L52)) |

### Admission Rules

An item is routed to the pending pool (instead of the active pool) when:

```
item.assert_height.is_some() && item.assert_height.unwrap() > current_height
```

This matches Chia's routing at [`mempool_manager.py:600-603`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/mempool_manager.py#L600).

### Rejection

The item is rejected with `PendingPoolFull` if either:

```
pending.len() >= config.max_pending_count
```

or:

```
pending_cost + item.virtual_cost > config.max_pending_cost
```

### Promotion

During `on_new_block(height, timestamp, ...)`, pending items whose timelocks are now satisfied (`assert_height <= height`) are extracted and returned in `RetryBundles.pending_promotions` for re-submission. Chia: `PendingTxCache.drain(new_peak.height)` ([`pending_tx_cache.py:91-108`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L91)).

## Acceptance Criteria

- [ ] Pending pool is a separate `HashMap<Bytes32, Arc<MempoolItem>>`.
- [ ] Items with `assert_height > current_height` are routed to the pending pool.
- [ ] Count limit `max_pending_count` (default 3000) is enforced.
- [ ] Cost limit `max_pending_cost` is enforced.
- [ ] Exceeding either limit returns `PendingPoolFull`.
- [ ] `on_new_block()` extracts promotable items and returns them in `RetryBundles.pending_promotions`.
- [ ] Pending pool uses its own lock (`pending_lock`).

## Implementation Notes

- **Primary file:** `src/pools/pending.rs`
- **Chia comparison:** Chia's `PendingTxCache` evicts highest-`assert_height` items first when full ([`pending_tx_cache.py:77-89`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L77)). dig-mempool rejects outright with `PendingPoolFull` for simplicity.
- **Re-submission:** Promoted items are returned as `SpendBundle` (not `MempoolItem`) because they need re-validation with fresh coin records and the current chain state.
- **pending_coin_index:** Used by [POL-005](POL-005.md) for pending-vs-pending conflict detection.
