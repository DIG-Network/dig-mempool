> **Authoritative requirement:** [POL-009](../NORMATIVE.md#POL-009)

## Summary

The singleton tracking index maps each singleton's `launcher_id` to an ordered list of bundle IDs that spend sequential versions of that singleton. This enables the mempool to track singleton fast-forward chains and ensures correct ordering during block selection.

A singleton is a coin that, when spent, creates exactly one child coin with a specific puzzle structure (singleton top-layer puzzle wrapping an inner puzzle). Singletons are identified by a `launcher_id` that remains constant across all versions. Multiple sequential spends of the same singleton can coexist in the mempool; the block builder rebases each spend's lineage proof to reference the previous spend's output.

This corresponds to Chia's `BundleCoinSpend.latest_singleton_lineage` ([`mempool_item.py:35-42`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/types/mempool_item.py#L35)) and `SingletonFastForward` tracking ([`eligible_coin_spends.py`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/eligible_coin_spends.py)).

## Specification

### Data Structure

```rust
// Singleton tracking (protected by pool_lock: RwLock, part of active pool state)
singleton_spends: HashMap<Bytes32, Vec<Bytes32>>
// Key: launcher_id (immutable singleton identity)
// Value: Vec of bundle_ids in lineage order (oldest version first)
```

### Insertion

When a new item with `singleton_lineage` is admitted:

1. Check `singleton_spends[launcher_id]` for existing entries.
2. **Sequential update:** If the new item spends the coin created by the latest existing item for this launcher_id, append the new item's bundle_id to the vector. This is not a conflict -- it is a sequential singleton update. Chia: `check_removals()` FF special case ([`mempool_manager.py:274-276`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool_manager.py#L274)).
3. **Conflicting update:** If the new item spends an older version (not the latest), treat as a conflict. Apply RBF rules. If RBF succeeds, replace the chain from the conflicting point onward.
4. **Fresh singleton:** If no existing items for this launcher_id, create a new entry with `vec![bundle_id]`.

### Removal

When an item with `singleton_lineage` is removed from the active pool:

1. Remove the item's bundle_id from `singleton_spends[launcher_id]`.
2. If the removed item was in the middle of a chain, all subsequent items in the chain become invalid (their input coins no longer exist). Cascade-evict them.
3. If the vector becomes empty, remove the launcher_id key.

### Block Selection Integration

During block selection ([SEL-008](../../selection/specs/SEL-008.md)), singleton chains are treated as dependency chains:

- All items in a singleton chain are included together or not at all.
- They are ordered by lineage (oldest first) in the output.
- The total cost is the sum of all items in the chain.

### Configuration

Singleton tracking is enabled/disabled via `MempoolConfig.enable_singleton_ff` (default: true).

## Acceptance Criteria

- [ ] `singleton_spends` is a `HashMap<Bytes32, Vec<Bytes32>>`.
- [ ] Keys are `launcher_id` values.
- [ ] Values are ordered lists of bundle IDs in lineage order.
- [ ] Sequential singleton updates append to the chain.
- [ ] Conflicting singleton updates trigger RBF.
- [ ] Removal cascade-evicts subsequent chain items.
- [ ] Block selection orders singleton chains by lineage (oldest first).
- [ ] The feature is gated by `enable_singleton_ff`.

## Implementation Notes

- **Primary file:** `src/pools/singleton.rs` or `src/admission/singleton.rs`
- **Detection:** Singleton detection is delegated to chia-consensus (`ELIGIBLE_FOR_FF` flag) and chia-sdk-driver (`SingletonLayer::parse_puzzle()`). The mempool does not parse singleton puzzles. See SPEC Section 11.2.
- **Lineage rebasing:** The mempool does NOT perform lineage rebasing. This is block production logic. The mempool returns items in correct lineage order; the caller uses `fast_forward_singleton()` from chia-consensus.
- **Chia comparison:** Chia handles FF rebasing during `new_peak()` ([`mempool_manager.py:921-937`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool_manager.py#L921)) and block generation ([`mempool.py:596-600`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool.py#L596)).

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| Fresh singleton added | Unit test | Singleton with new launcher_id | singleton_spends[launcher_id] = [bundle_id] |
| Sequential update appends | Unit test | Second spend of same singleton, spending latest output | Vector becomes [id1, id2] |
| Conflicting update triggers RBF | Unit test | Spend older version of same singleton | RBF rules applied |
| Removal cascade-evicts chain | Unit test | Remove middle item from chain [A, B, C] | B and C removed |

| Empty chain cleaned up | Unit test | Remove only item for a launcher_id | launcher_id key removed |
