# POL-008: Identical Spend Dedup Index

> **Authoritative requirement:** [POL-008](../NORMATIVE.md#POL-008)

## Summary

The identical spend dedup index tracks cases where multiple bundles contain the same `CoinSpend` (same coin, same puzzle, same solution). When two bundles spend the same coin with the same solution, the CLVM execution result is identical. The block builder can run the puzzle once and reuse the result, reducing effective block cost. The mempool's role is limited to tracking dedup eligibility and adjusting cost accounting so that block selection makes better decisions.

This corresponds to Chia's `IdenticalSpendDedup` ([`eligible_coin_spends.py`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/eligible_coin_spends.py)) and the `ELIGIBLE_FOR_DEDUP` flag mechanism. Per-spend cost is tracked in `BundleCoinSpend.cost` ([`mempool_item.py:32`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/types/mempool_item.py#L32)).

## Specification

### Data Structure

```rust
// Dedup index (protected by pool_lock: RwLock, part of active pool state)
dedup_index: HashMap<(Bytes32, Bytes32), Bytes32>
// Key: (coin_id, sha256(solution))
// Value: bundle_id of the first bundle that established the cost
```

### Insertion

When a new item with `eligible_for_dedup == true` is admitted to the active pool:

1. For each `CoinSpend` in the bundle, compute the key `(coin_id, sha256(solution))`.
2. Check `dedup_index` for an existing entry with the same key.
3. If found: the CLVM cost for this specific spend is NOT added to the new item's effective cost. Record the cost saving.
4. If not found: insert the key with the new item's bundle ID as the cost-bearer.

### Eligibility

A bundle is `eligible_for_dedup` only if ALL spends in the bundle have the `ELIGIBLE_FOR_DEDUP` (0x1) flag set in `OwnedSpendConditions.flags`. This flag is set by chia-consensus's `MempoolVisitor` during CLVM execution when `MEMPOOL_MODE` is active. The mempool reads the flag; it does not check canonical encoding itself.

### Cost Adjustment

```
cost_saving = sum of CLVM execution cost for each deduplicated spend
effective_cost = item.cost - cost_saving
effective_virtual_cost = effective_cost + (num_spends * SPEND_PENALTY_COST)
```

The cost saving affects the item's effective cost during block selection ([SEL-003](../../selection/specs/SEL-003.md) through [SEL-006](../../selection/specs/SEL-006.md)), allowing more transactions to fit within the block cost budget.

### Invalidation

When the "first bundle" (the cost-bearer) is removed from the active pool:

1. Find the next item with the same dedup key.
2. Transfer cost establishment to that item.
3. Re-compute affected items' effective costs and fee rates.
4. Update `dedup_index` to point to the new cost-bearer.

### Configuration

Dedup is enabled/disabled via `MempoolConfig.enable_identical_spend_dedup` (default: true).

## Acceptance Criteria

- [ ] `dedup_index` is a `HashMap<(Bytes32, Bytes32), Bytes32>`.
- [ ] Keys are `(coin_id, sha256(solution))`.
- [ ] Values are the bundle ID of the cost-bearer.
- [ ] Duplicate spends reduce the new item's effective cost.
- [ ] Only items with `eligible_for_dedup == true` participate.
- [ ] Cost-bearer removal triggers re-assignment.
- [ ] The feature is gated by `enable_identical_spend_dedup`.

## Implementation Notes

- **Primary file:** `src/pools/dedup.rs` or `src/admission/dedup.rs`
- **Solution hash:** Compute `sha256(solution_bytes)` from the `CoinSpend.solution` field. This is a raw hash, not a tree hash.
- **Chia comparison:** Chia's dedup check happens during block building (`eligible_coin_spends.py`), not during admission. dig-mempool moves the cost accounting to admission time for better selection decisions.
- **Canonical encoding:** The `ELIGIBLE_FOR_DEDUP` flag requires canonical CLVM encoding (no back-references, minimal-length atoms). This is enforced by chia-consensus, not the mempool. See SPEC Section 12.2.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| First bundle establishes cost | Unit test | Bundle A with dedup-eligible spend of coin X | dedup_index[(X, hash)] = A.bundle_id |
| Second bundle gets cost saving | Unit test | Bundle B with same spend of coin X | B.cost_saving > 0 |
| Non-eligible not indexed | Unit test | Bundle with `eligible_for_dedup = false` | Not added to dedup_index |
| Cost-bearer removal re-assigns | Unit test | Remove bundle A, bundle B has same spend | dedup_index updated to B.bundle_id |
| Feature disabled | Unit test | Set `enable_identical_spend_dedup = false` | No dedup tracking |
| Different solutions not deduped | Unit test | Same coin, different solution | Separate dedup_index entries |
| Effective cost calculation | Unit test | Bundle with 2 spends, 1 deduped | effective_cost = total_cost - deduped_spend_cost |
| Selection uses effective cost | Integration test | Deduped bundle fits budget where non-deduped would not | Bundle selected |
