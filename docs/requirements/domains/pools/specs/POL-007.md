# POL-007: Seen Cache

> **Authoritative requirement:** [POL-007](../NORMATIVE.md#POL-007)

## Summary

The seen cache is an LRU-bounded set of bundle IDs that have been submitted to the mempool, regardless of whether they were accepted or rejected. Bundle IDs are added to the seen cache **before** CLVM validation begins. This prevents repeated expensive CLVM dry-runs for the same (potentially invalid) bundle, providing DoS protection.

This corresponds to Chia's `seen_bundle_hashes` set in `MempoolManager` ([`mempool_manager.py:298`](https://github.com/Chia-Network/chia-blockchain/blob/main/chia/full_node/mempool_manager.py#L298)), which uses the same pre-validation semantics.

## Specification

### Data Structure

```rust
// Seen cache (protected by seen_lock: RwLock)
seen_cache: LruSet<Bytes32>  // or equivalent bounded set
```

The seen cache is a set (not a map) -- it only tracks whether a bundle ID has been seen, not any associated data.

### Capacity

| Parameter | Default | Source |
|-----------|---------|--------|
| `max_seen_cache_size` | 10,000 | `MempoolConfig.max_seen_cache_size`. Chia: hardcoded in `MempoolManager`. |

When the cache is full, the least-recently-used (oldest) entry is evicted to make room for the new entry.

### Insertion Timing

Bundle IDs are added to the seen cache at the **beginning** of the submission pipeline, in Phase 1, before CLVM validation. The sequence is:

1. Compute `bundle_id = bundle.name()`.
2. Check if `bundle_id` is in `seen_cache`, active pool, pending pool, or conflict cache.
3. If found anywhere, return `MempoolError::AlreadySeen(bundle_id)`.
4. **Add `bundle_id` to `seen_cache`** (even before knowing if validation will succeed).
5. Proceed with CLVM validation.

This means a bundle that fails CLVM validation is still marked as "seen" and will be rejected immediately on re-submission without incurring another CLVM dry-run.

### Not Included in Snapshots

The seen cache is ephemeral and is NOT included in `MempoolSnapshot`. After a snapshot/restore cycle, previously-seen bundles may be re-validated once. This is acceptable because the cost is bounded by the mempool's capacity.

### Clearing

The seen cache is cleared by `clear()` (reorg recovery). It is NOT cleared by `on_new_block()`.

---

## Acceptance Criteria

- [ ] Seen cache is an LRU-bounded set with capacity `max_seen_cache_size`.
- [ ] Default capacity is 10,000.
- [ ] Bundle IDs are added before CLVM validation begins.
- [ ] Re-submission of a seen bundle returns `AlreadySeen` without CLVM execution.
- [ ] The cache also checks against active pool, pending pool, and conflict cache.
- [ ] LRU eviction removes the oldest entry when at capacity.
- [ ] Seen cache is NOT included in snapshot/restore.
- [ ] `clear()` resets the seen cache.

## Implementation Notes

- **Primary file:** `src/pools/seen.rs` or inline in `src/mempool.rs`
- **LRU implementation:** Use `std::collections::LinkedHashMap` or a purpose-built LRU set. The `lru` crate is a common Rust LRU implementation.
- **Chia comparison:** Chia uses a Python set with manual size management. dig-mempool uses a proper LRU for bounded memory.
- **Locking:** The seen cache is checked in Phase 1 (before the write lock is acquired). It uses `seen_lock: RwLock` -- read lock for checking, write lock for insertion.
- **Phase 1 race:** Two concurrent submissions of the same bundle may both pass the seen check before either inserts. This is handled by re-checking in Phase 2 under the write lock ([SPEC Section 5.1 step 6](../../../../resources/SPEC.md)).

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| First submission passes | Unit test | Submit bundle for the first time | Not rejected by seen cache |
| Second submission rejected | Unit test | Submit same bundle twice | Second returns `AlreadySeen` |
| Added before validation | Unit test | Submit invalid bundle, then re-submit | Second returns `AlreadySeen` (no CLVM re-execution) |
| LRU eviction | Unit test | Fill cache to 10000, add one more | Oldest entry evicted; submitting evicted bundle succeeds |
| Active pool check | Unit test | Bundle in active pool, not in seen cache | Returns `AlreadySeen` |
| Pending pool check | Unit test | Bundle in pending pool | Returns `AlreadySeen` |
| Not in snapshot | Unit test | Snapshot and restore, re-submit previously seen bundle | Passes seen check (re-validated) |
| Clear resets cache | Unit test | Add entries, call `clear()`, re-submit | Passes seen check |
