# POL-010: Concurrency

> **Authoritative requirement:** [POL-010](../NORMATIVE.md#POL-010)

> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The mempool uses fine-grained locking to enable concurrent reads and minimize write contention. Each pool/cache has its own lock, and the expensive CLVM validation (Phase 1) runs without holding any write lock. The `Mempool` struct must be `Send + Sync` to be safely shared across async tasks and threads.

This is an improvement over Chia L1, which holds a single lock for the entire validation + insertion flow. dig-mempool's two-phase design (SPEC Section 5.1) enables concurrent CLVM validation while only holding the write lock for fast HashMap operations during Phase 2.

## Specification

### Lock Table

| Lock | Protects | Type | Hot Path |
|------|----------|------|----------|
| `pool_lock` | `items`, `coin_index`, `mempool_coins`, `dependencies`, `dependents`, `singleton_spends`, `dedup_index`, accumulators | `RwLock` | Phase 2 (write), `select_for_block` (read), queries (read) |
| `pending_lock` | `pending`, `pending_coin_index`, `pending_cost` | `RwLock` | Phase 2 for timelocked items |
| `conflict_lock` | `conflict_cache`, `conflict_cost` | `RwLock` | Phase 2 on conflict |
| `seen_lock` | `seen_cache` | `RwLock` | Phase 1 (dedup check: read), Phase 1 (insert: write) |
| `bls_lock` | `bls_cache` | `Mutex` | Phase 1 (BLS signature verification via dig-clvm) |
| `fee_lock` | `fee_tracker` | `RwLock` | `on_new_block` (write), `estimate_fee_rate` (read) |
| `hooks_lock` | `hooks` | `RwLock` | Under parent write lock (Phase 2), `add_event_hook` (write) |

### Locking Discipline

1. **Phase 1 (lock-free for pool_lock):** CLVM validation via `dig_clvm::validate_spend_bundle()` does NOT hold `pool_lock`. It may acquire `seen_lock` (read then write) and `bls_lock` (mutex for BLS cache).

2. **Phase 2 (write lock):** After validation, acquire `pool_lock` write lock for state mutation. All Phase 2 operations are fast HashMap inserts/removes -- no CLVM execution under this lock.

3. **Lock ordering:** When multiple locks are needed, acquire in this order to prevent deadlocks:
   - `pool_lock` -> `pending_lock` -> `conflict_lock`
   - Never hold `bls_lock` while holding `pool_lock`.
   - `seen_lock` is independent (only held briefly in Phase 1).

4. **Read concurrency:** `select_for_block()`, queries (`get`, `contains`, `stats`), and `estimate_fee_rate()` only acquire read locks, allowing full concurrency with other reads.

### Send + Sync

The `Mempool` struct MUST implement `Send + Sync`. This requires:
- All fields are `Send + Sync` or wrapped in `Send + Sync` containers.
- `Arc<MempoolItem>` is `Send + Sync` (since `MempoolItem` is `Send + Sync`).
- `RwLock` and `Mutex` from `std::sync` are `Send + Sync`.
- No `Rc`, `Cell`, or `RefCell` in the public API (use `Arc`, `AtomicU64`, `Mutex` instead).

### Interior Mutability

The `Mempool` struct uses interior mutability via locks. All public methods take `&self`, not `&mut self`. This matches SPEC Section 1.3 Decision 1: "Interior mutability (`&self` + internal `RwLock`)."

## Acceptance Criteria

- [ ] `pool_lock` is an `RwLock` protecting active pool state.
- [ ] `pending_lock` is an `RwLock` protecting pending pool state.
- [ ] `conflict_lock` is an `RwLock` protecting the conflict cache.
- [ ] `seen_lock` is an `RwLock` protecting the seen cache.
- [ ] `bls_lock` is a `Mutex` protecting the BLS cache.
- [ ] The `Mempool` struct is `Send + Sync`.
- [ ] CLVM validation (Phase 1) does not hold `pool_lock`.
- [ ] Lock ordering prevents deadlocks.

## Implementation Notes

- **Primary file:** `src/mempool.rs` (struct definition with lock fields)
- **RwLock choice:** `std::sync::RwLock` is sufficient. `parking_lot::RwLock` is a potential optimization for reduced overhead.
- **Chia comparison:** Chia uses Python's GIL and an explicit lock (`MempoolManager.lock`) for the entire flow. dig-mempool's fine-grained locking is listed as improvement #1 in SPEC Section 1.4.
- **Batch submission:** `submit_batch()` acquires `pool_lock` write once for all insertions, amortizing lock acquisition cost.
- **Static assertion:** Use `static_assert_impl_all!(Mempool: Send, Sync)` or equivalent compile-time check.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| Send + Sync compile check | Unit test | `fn assert_send_sync<T: Send + Sync>() {} assert_send_sync::<Mempool>();` | Compiles |
| Concurrent reads | Integration test | Multiple threads call `get()` and `select_for_block()` simultaneously | No deadlock, correct results |
| Concurrent submit + read | Integration test | One thread submits, another reads | No deadlock |
| Phase 1 does not hold pool_lock | Integration test | Submit bundle; verify pool_lock read is available during CLVM validation | Read succeeds during Phase 1 |
| Lock ordering respected | Integration test | Concurrent operations touching multiple locks | No deadlock after sustained load |
| Interior mutability | Unit test | All public methods accept `&self` | Compiles with shared reference |
| BLS cache mutex | Unit test | Two concurrent submissions both need BLS verification | Serialized through Mutex, both succeed |
| Batch single lock acquisition | Unit test | Batch submit 10 bundles, verify single pool_lock write | Items inserted atomically |
