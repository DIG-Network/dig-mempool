## Summary

The pending pool maintains a `pending_coin_index` that maps each spent coin ID to the pending bundle that spends it. When a new timelocked item is being admitted to the pending pool, this index is checked for conflicts with existing pending items. Conflicts between pending items are resolved via Replace-By-Fee (RBF) rules, identical to active pool RBF.

This is a dig-mempool improvement over Chia L1. Chia's `PendingTxCache` does not perform conflict detection among pending items ([`pending_tx_cache.py:50-111`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L50)). dig-mempool adds pending-vs-pending dedup to prevent the pending pool from accumulating conflicting bundles that will all fail when promoted.

## Specification

### Data Structure

```rust
pending_coin_index: HashMap<Bytes32, Bytes32>  // spent_coin_id -> pending bundle_id
```

This index is maintained alongside the `pending` HashMap. On insertion, every coin in `item.removals` is added. On removal, every coin is cleaned up.

### Conflict Detection

When a new timelocked item is being routed to the pending pool:

1. For each coin in `new_item.removals`, check `pending_coin_index`.
2. If any coin is found, collect the conflicting pending bundle IDs.
3. Apply RBF rules (identical to active pool RBF from SPEC Section 5.11):
   a. **Superset rule:** Every coin in the conflicting pending item(s) must also be in the new item's removals.
   b. **Higher FPC:** `new_item.fee_per_virtual_cost_scaled > conflicting_fees / conflicting_cost`.
   c. **Minimum fee bump:** `new_item.fee >= conflicting_fees + MIN_RBF_FEE_BUMP`.
4. If RBF succeeds, remove the conflicting pending items and admit the new item.
5. If RBF fails, reject the new item (it does not go to the conflict cache -- conflict caching is only for active pool conflicts).

### Invariants

- Every coin in every pending item's `removals` has an entry in `pending_coin_index`.
- No two pending items spend the same coin (maintained by RBF resolution).

---

## Acceptance Criteria

- [ ] `pending_coin_index` is maintained for all pending items.
- [ ] New pending items are checked against `pending_coin_index` for conflicts.
- [ ] Pending-vs-pending conflicts use the same RBF rules as active pool conflicts.
- [ ] Successful RBF removes the conflicting pending item(s) before inserting the new one.
- [ ] Failed RBF rejects the new item without adding it to the conflict cache.
- [ ] The index is cleaned up when pending items are removed (promotion or eviction).

---

## Implementation Notes

- **Primary file:** `src/pools/pending.rs`
- **RBF reuse:** The RBF logic should be factored into a shared function used by both active pool conflict resolution and pending pool dedup.
- **Performance:** O(r) per submission where r is the number of removals in the new item. Typically r < 10.
- **Chia comparison:** Chia does NOT perform pending-vs-pending dedup. This is listed as improvement #4 in SPEC Section 1.4.

## Verification

### Test Plan

| Test | Type | Description | Expected Result |
|------|------|-------------|-----------------|
| Index populated on insert | Unit test | Add pending item with removals [A, B] | pending_coin_index[A] and [B] point to bundle_id |
| Conflict detected | Unit test | Two pending items both spending coin X | Second item triggers conflict |
| RBF succeeds | Unit test | New item has higher FPC, superset of coins | Old item replaced, new item admitted |
| RBF fails: not superset | Unit test | New item spends coin X, old item spends X and Y, new doesn't spend Y | Rejected (not superset) |
| RBF fails: FPC too low | Unit test | New item has lower FPC than existing | Rejected |
| RBF fails: fee bump too low | Unit test | Fee bump < MIN_RBF_FEE_BUMP | Rejected |
| Index cleaned on promotion | Unit test | Promote pending item via on_new_block | pending_coin_index entries removed |
| No conflict cache for pending | Unit test | Pending RBF fails | Item NOT added to conflict cache |
