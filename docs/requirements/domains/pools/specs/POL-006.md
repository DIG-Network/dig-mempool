# POL-006: Conflict Cache

## Summary

The conflict cache stores bundles that failed RBF against an existing active pool item. When the conflicting active item is later confirmed in a block (and removed by `on_new_block()`), the cached bundles are returned for re-submission, giving them a second chance at admission.

This corresponds to Chia's `ConflictTxCache` ([`pending_tx_cache.py:13-47`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L13)), which stores conflicting transactions with count and cost limits.

The conflict cache stores raw `SpendBundle` objects (not `MempoolItem`) because the cached bundles need full re-validation when retried -- coin records may have changed.

## Specification

### Data Structure

```rust
// Conflict cache storage (protected by conflict_lock: RwLock)
conflict_cache: HashMap<Bytes32, SpendBundle>  // bundle_id -> original spend bundle
conflict_cost:  u64                            // aggregate cost (estimated)
```

### Capacity Limits

| Parameter | Default | Source |
|-----------|---------|--------|
| `max_conflict_count` | 1,000 | `MempoolConfig.max_conflict_count`. Chia: `ConflictTxCache._cache_max_size = 1000` ([`pending_tx_cache.py:15`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L15)) |
| `max_conflict_cost` | 1x block cost | `MempoolConfig.max_conflict_cost`. Chia: `ConflictTxCache._cache_max_total_cost` ([`pending_tx_cache.py:14`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L14)) |

### Insertion

A bundle is added to the conflict cache when:

1. It conflicts with one or more active pool items (shared spent coins).
2. RBF evaluation fails (the new bundle cannot replace the existing one).
3. The conflict cache has capacity (count and cost limits not exceeded).

Chia adds to the conflict cache at [`mempool_manager.py:595-599`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/mempool_manager.py#L595).

If the conflict cache is full, the bundle is silently dropped (not stored). No error is returned beyond the original `Conflict` error.

### Retrieval (Drain)

During `on_new_block()`, all conflict cache entries are drained and returned as `RetryBundles.conflict_retries`. The caller re-submits them with fresh coin records. Chia: `ConflictTxCache.drain()` ([`pending_tx_cache.py:40-44`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L40)).

### What Does NOT Go to Conflict Cache

- Cascade-evicted items (their input coins no longer exist).
- Pending pool RBF failures.
- Items rejected for non-conflict reasons (fee too low, cost exceeded, etc.).

---

## Acceptance Criteria

- [ ] Conflict cache is a `HashMap<Bytes32, SpendBundle>`.
- [ ] Count limit `max_conflict_count` (default 1000) is enforced.
- [ ] Cost limit `max_conflict_cost` is enforced.
- [ ] Bundles that fail active-pool RBF are stored in the conflict cache.
- [ ] `on_new_block()` drains all entries and returns them as `conflict_retries`.
- [ ] Cascade-evicted items do NOT go to the conflict cache.
- [ ] Conflict cache uses its own lock (`conflict_lock`).

---

## Implementation Notes

- **Primary file:** `src/pools/conflict.rs`
- **SpendBundle not MempoolItem:** The cache stores raw `SpendBundle` because re-submission requires full re-validation (CLVM execution, coin record lookup, conflict check against the updated active pool).
- **Chia comparison:** Chia's `ConflictTxCache` also stores the `SpendBundle` ([`pending_tx_cache.py:33`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/full_node/pending_tx_cache.py#L33)) and drains all entries on new block.
- **cost estimation:** `conflict_cost` tracks an estimated aggregate cost. Since we store raw bundles (not validated items), the cost is estimated from the original validation. This is acceptable because it is only used for capacity bounding.

| Count limit enforced | Unit test | Fill cache to 1000, add one more | New bundle silently dropped |
